## Stage 113 — fix: postgres_named_query — drop specifyInputSchema, trigger back to passthrough (proven on tgbot15 with the operator sending a real message)

| PR | Type | Summary |
|---|---|---|
| TBD | fix | Stages 110/111/112 all chased the wrong path: with `executeWorkflowTrigger` in `workflowInputs` mode + the `toolWorkflow` node carrying any flavour of `specifyInputSchema` (`manual` JSON Schema with enum, or `fromJson` example), n8n 1.121's bridge between LLM args and the trigger's `$json` shipped `query_name: null` regardless of what the LLM provided. Verified deterministically by two diagnostic rounds on `factory-test-tgbot14` (DEBUG1, after Stage 111's `workflowInputs`) and on `factory-test-tgbot15` (DEBUG2, after Stage 112's `schemaType:"fromJson"`): both surfaced identical evidence — `Keys at root: ["query_name"]`, `typeof query_name: object` (JS quirk: `typeof null === "object"`), `_raw_input: {"query_name": null}`. That's the first end-to-end success for the tool. Fix in main: (1) `templates/system/workflows/n8n/postgres-named-queries.json` — trigger back to `inputSource:"passthrough"` (the `workflowInputs` config was load-bearing only for Stages 110-112's schema dance, now reverted), Normalize Code node updated to extract `query_name` from `$json.query` as a raw string against the 4-value whitelist (also tries JSON.parse for safety). (2) `unknown-agent.json` + `ops-agent.json` — `postgres_named_query` toolWorkflow nodes lose `specifyInputSchema`, `schemaType`, `jsonSchemaExample`. Inline code-comment on the Normalize node records the 3-stage diagnostic so a future reader sees why this specific shape was chosen. ## Stage 112 — fix: postgres_named_query schemaType — Stage 111's diagnostic on tgbot14 proved the LLM was passing `query_name: null`

| PR | Type | Summary |
|---|---|---|
| TBD | fix | Stage 111's Normalize Code node + `_raw_input` debug fields wasn't surfaced by the LLM in its summary replies, so the diagnostic was inconclusive on `tgbot13`. Pushed a temporary diagnostic to `factory-test-tgbot14` (`Send Debug to Telegram` node after `Unknown Query`, hardcoded credential id `OIFaNWm8pQYzOj2s`) that **bypassed the LLM entirely** and surfaced the raw `$json` shape directly to the operator's Telegram. **Result**: `Keys at root: ["query_name"]`, `Item count: 1`, `_raw_input: {"query_name": null}` — the trigger IS exposing `query_name` at root level (Stage 110's `workflowInputs` fix was correct), but the **value arriving is `null`**, not the LLM's chosen string. Root cause: `toolWorkflow` v2.1 with `specifyInputSchema:true, schemaType:"manual"` and a JSON-Schema-with-enum-string. n8n's parser doesn't reliably convert string `enum` constraints to a Zod schema the LLM provider can act on — either the LLM never receives a useful schema (so it omits the parameter, and the trigger fills the missing field with `null` defaults from `workflowInputs.values=[{name:"query_name",type:"string"}]`), or n8n's manual-schema → Zod translation drops the enum's string values and substitutes null on validation. Fix: switch the two `postgres_named_query` `toolWorkflow` nodes (in `unknown-agent.json` and `ops-agent.json`) from `schemaType:"manual"` + JSON-Schema-string to `schemaType:"fromJson"` + a `jsonSchemaExample` literal (`{"query_name":"style_profile_get"}` in unknown, `{"query_name":"executions_summary_24h"}` in ops). The "fromJson" path is n8n's documented well-trodden flow — it derives the Zod schema from the example value, and the example serves as a worked illustration to the LLM of exactly what to pass. Tool description also tightened to be explicit: "**MUST** pass `query_name` as one of EXACTLY these 4 string values". No template change to `postgres-named-queries.json` itself — Stage 111's Normalize node stays in place and the trigger's `workflowInputs` stays at `query_name:string`. Per CLAUDE.md `templates/system/` edits reach only newly-provisioned systems; verification = a fresh `factory-test-tgbot15` (the tgbot14 diagnostic doesn't carry the fromJson fix — it lives in its own repo's copy of the template, frozen at provision time). |

## Stage 111 — fix: postgres-named-queries needs a defensive Normalize Code node — Stage 110's workflowInputs fix alone wasn't enough

| PR | Type | Summary |
|---|---|---|
| TBD | fix | Stage 110 changed `templates/system/workflows/n8n/postgres-named-queries.json`'s `executeWorkflowTrigger` from `inputSource:"passthrough"` to `inputSource:"workflowInputs"` with `values:[{name:"query_name",type:"string"}]`, hoping the LLM's `query_name` would land at `$json.query_name`. Live verification on `factory-test-tgbot13` shows that didn't happen: 3 postgres-named-queries executions (22, 26, 27) — each ~10-30 ms — all routed straight to the Switch's `Unknown Query` fallback, never touching any Postgres branch. The LLM is calling the tool, but the args still aren't reaching `$json` in the shape the Switch expects (either `workflowInputs` isn't surfacing them on this n8n version + toolWorkflow v2.1 combination, or the args arrive at a different path like `$json.input.*`, `$json.query.*`, or `$json.arguments`). Fix without further blind guessing: new **Normalize Input** Code node between trigger and Switch — extracts `query_name` defensively from any of the observed shapes (`raw.query_name`, `raw.input.query_name`, `raw.query.query_name`, `JSON.parse(raw.query).query_name`, `JSON.parse(raw.arguments).query_name`), and **always emits `_raw_input: <the original $json>`** so the next test run will reveal the actual shape via the `Unknown Query` Set node's new `_debug_received` field if the normalization still misses. Switch now reads `={{ $json.query_name }}` from Normalize Input's output (a no-op rename — `$json` is just the previous node's data); Format Output reads `$('Normalize Input').first().json.query_name`; Unknown Query reads the same and adds `_debug_received` to its response payload. Idempotent + soft-fail: when the LLM actually provides `query_name` correctly at any known path, the Switch matches and the right Postgres query runs; when it doesn't, the LLM gets an error response with the raw payload it sent — much easier to diagnose than the current "Unknown Query" black box. Per CLAUDE.md `templates/system/` edits reach only newly-provisioned systems; verification = same 3 Telegram messages on a fresh test (or a configure re-run on a system whose template would update, though tgbot13's already-installed subworkflow won't auto-update). |
