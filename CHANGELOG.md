## Stage 19 — deploy-plane: single-redeploy env upsert + n8n HTTP readiness gate

| PR | Type | Summary |
|---|---|---|
| TBD | fix | Two compounding fixes to `templates/system/.github/workflows/deploy-railway-cloudflare.yml` that made redeploys flaky — a deploy could fail at step 8 "Set up n8n owner account" with `GET /rest/settings` HTTP 404/403 and then pass on an immediate re-dispatch (observed on factory-test-23: run 26331335520 failed → 26331473093 passed). **(A) Collapse the env-var redeploy storm.** `_upsert` issued one Railway `variableUpsert` per variable and step 5 fired 15 in parallel for n8n + 5 for Postgres; the old comment claimed the parallel `&` fan-out "lands a single redeploy" — it does not, each `variableUpsert` triggers its own redeploy (proven: `list_railway_deployments` showed **12 n8n deployments created within ~4 s** on one run, 11 superseded). Replaced with `_upsert_collection` → one `variableCollectionUpsert` per service (all vars in a single `EnvironmentVariables` JSON map; schema verified by introspection: `VariableCollectionUpsertInput{projectId!,environmentId!,serviceId,variables!,replace,skipDeploys}`), so each service redeploys once instead of ~15×. Railway reference values (`${DLR}{{Postgres.X}}`) are preserved verbatim inside the jq map. **(B) HTTP readiness gate.** Step 7 only verifies the TLS cert (`_probe_tls_cn`), which does not imply the edge routes HTTP to a healthy n8n — after a redeploy the edge can return 404/403 ("Host not in allowlist") for minutes while the custom domain re-propagates, and step 8's single no-retry `curl /rest/settings` raced that window. Replaced with a bounded poll loop (36×10 s ≈ 6 min) that waits for HTTP 200 before proceeding, mirroring the cert-wait loop; still fails fast after the cap so a genuinely crash-looping n8n is not masked. The notifier (step 9) runs right after step 8, so it inherits the gate. |

Template edits propagate only to newly-provisioned systems (per CLAUDE.md). factory-test-23 gets the same two edits in its frozen copy so its own redeploys are reliable; other existing systems are not backfilled.

## Stage 17 — deploy-plane: notifier workflow create missing `active`

| PR | Type | Summary |
|---|---|---|
| TBD | fix | Add `active: false` to the `POST /rest/workflows` create payload in the Stage 15 notifier step of `templates/system/.github/workflows/deploy-railway-cloudflare.yml`. n8n 1.121.0's internal create handler writes `req.body.active` straight into the `workflow_entity.active` column, which is `NOT NULL` with no DB default — so omitting the field made Postgres reject the insert and the endpoint returned `HTTP 500 {"message":"null value in column \"active\" of relation \"workflow_entity\" violates not-null constraint"}`. ## Stage 15 — deploy-plane: auto-create "n8n is ready" Telegram notifier workflow

| PR | Type | Summary |
|---|---|---|
| TBD | feature | Add a new step `Create "n8n is ready" Telegram notifier workflow` to `templates/system/.github/workflows/deploy-railway-cloudflare.yml`, inserted between `Set up n8n owner account` and `Persist Railway IDs to system SM`. On first deploy only, the step logs into the freshly provisioned n8n as the owner via `POST /rest/login` (capturing the `n8n-auth` JWT cookie), creates a `telegramApi` credential, POSTs a 2-node workflow (Webhook → Telegram sendMessage) to `/rest/workflows`, activates it, and fires the webhook once — which causes the n8n instance itself to send `🎉 n8n מוכן: <system>` with URL + login email. The workflow is named `factory-master: n8n ready notifier`; on redeploy the step detects the existing name and exits 0 immediately so the Telegram message never re-fires. Uses n8n's internal `/rest/*` API (the same API the prior `Set up n8n owner account` step uses two lines up) with cookie auth — no public-API key minting needed. Triple purpose: (a) out-of-band confirmation that n8n is end-to-end healthy (login + credentials + workflows + webhook listener + outbound HTTPS to api.telegram.org all work), (b) closes the gap where the operator had to manually open `https://n8n-<system>.or-infra.com` after deploy to confirm the new system actually serves, (c) leaves the operator with a working example workflow they can clone for their own automations. Telegram-node JSON shape (`typeVersion 1.2`, resource `message`, operation `sendMessage`, credential ref `{telegramApi:{id,name}}`) and credential field name `accessToken` confirmed against n8n source `packages/nodes-base/credentials/TelegramApi.credentials.ts`. Activation tries `POST /rest/workflows/{id}/activate` first and falls back to `PATCH /rest/workflows/{id}` with `{active:true}` so it survives an n8n minor-version bump. Login does the same fallback between modern `emailOrLdapLoginId` and legacy `email` field names. All API responses get their first 400 bytes printed on failure so operators can diagnose without re-running. Webhook node uses `responseMode: lastNode` so the `POST /webhook/<uuid>` HTTP response only returns after the Telegram node finishes — a 200 here is positive confirmation the Telegram API accepted the message, not just that the workflow started. |
| TBD | feature | **Per-system dedicated Telegram bot, not the factory-admin bot.** The notifier reads from two **new** per-system secrets — `n8n-telegram-bot-token` and `n8n-telegram-chat-id` — **distinct from** the factory-admin's `telegram-bot-token` / `telegram-chat-id` that `register-system-app.yml` uses out of `or-factory-master-control` SM. This keeps end-user n8n notifications on a system-owned bot (e.g. the operator's own BotFather bot for that system) instead of leaking system-status pings into the factory-admin channel. Two corresponding empty shells are pre-created by `.github/workflows/provision-system.yml`'s `Pre-create runtime secret shells` step (added to the `RUNTIME_SHELLS` array alongside the existing `n8n-*` / `railway-*` shells), so the deploy SA's per-secret `secretAccessor` grant is in place before the deploy step runs. The operator manually populates the shells via `gcloud secrets versions add n8n-telegram-bot-token --project=<system> --data-file=-` between provision and deploy. If either secret is still empty when deploy runs (operator forgot to fill them, or the system is being deployed before the bot is provisioned), the notifier step writes a clear ⚠️ block to `$GITHUB_STEP_SUMMARY` (with the exact `gcloud secrets versions add` commands to copy-paste) and exits 0 — deploy as a whole still succeeds, the operator can populate and redispatch later. The `copy-generic-secrets.sh` factory-bot secrets are NOT touched by this stage and continue to serve `register-system-app.yml`. |

## Stage 14 — deploy-plane: retry-with-recreate on TLS cert wait

| PR | Type | Summary |
|---|---|---|
| TBD | fix | Replace `Wait for Railway TLS cert` in `templates/system/.github/workflows/deploy-railway-cloudflare.yml:736-762` with a 2-attempt retry-with-recreate loop. Each attempt polls the TLS handshake for 5 min via the existing `openssl s_client` + cert-CN-match pipeline (unchanged); between attempts, force `customDomainDelete` + `customDomainCreate` (helpers lifted from the existing `Configure Cloudflare DNS:687-731` block) and re-publish the resulting CNAME target to Cloudflare via a freshly-minted 1h-expiry scoped DNS:Edit token (revoked immediately after use). Surfaced today on two consecutive systems (factory-test-20, factory-test-21) where Railway lost the first verification attempt — the prior 1-recreate + 5-min wait passed silently, the next step (`Set up n8n owner account`) curl'd into the missing cert, and the Stage 9 fail-fast (now archived) triggered on a non-200 HTTP that looked like an n8n bug rather than a Railway timing issue. Critical observation: between factory-test-21's failed Deploy #1 and successful Deploy #2, the customDomain sat in `verified=false` for 9 idle minutes with no change (verified directly via two `inspect_railway_service_direct` MCP calls) — direct evidence that wall-clock time alone does not unstick Railway's verification, but a fresh `customDomainDelete` + `customDomainCreate` does. Together with the Stage 9 fail-fast, this forms a single-dispatch single-success contract when Railway is healthy, and a fast-fail with an operator-actionable error when Railway is throttled. |
