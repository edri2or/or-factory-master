## Stage 110 — fix: 3 bugs from Stage 108 live verification — postgres_named_query input shape, sub-agent persona leaks, ops-agent tool gap

| PR | Type | Summary |
|---|---|---|
| TBD | fix | Surfaced during Stage 108's live verification on `factory-test-tgbot12` with 3 real Telegram messages (the discipline the handoff §3 rule 3 prescribed — "install PASS is not proof"). Three distinct bugs found, all fixed in one PR because they were found in the same verification round. **(1) postgres_named_query input shape — Stage 108 bug.** `templates/system/workflows/n8n/postgres-named-queries.json`'s `executeWorkflowTrigger` was set to `inputSource:"passthrough"`, but n8n's `toolWorkflow` v2.1 with `useSchema=true` reads the LLM's args via `getCurrentWorkflowInputData()` (in `packages/@n8n/nodes-langchain/nodes/tools/ToolWorkflow/v2/utils/WorkflowToolService.ts`), which expects `inputSource:"workflowInputs"` with explicit field defs. With `passthrough` the function returns empty → the LLM's `query_name` never reaches `$json` → the Switch reads undefined and falls to `Unknown Query` → tool returns `{ok:false, error:"unknown query_name…"}`. Caught on tgbot12 executions 37 and 38 (both `success`, both routed to `Unknown Query`). Fix: trigger now `inputSource:"workflowInputs"` + `workflowInputs:{values:[{name:"query_name",type:"string"}]}`. Zod schema on the toolWorkflow side keeps enforcing the enum at the LLM/tool boundary. **(2) Sub-agent persona leak — Stage 106 incomplete + false-positive verification.** ops/code/research/infra agents all carry an explicit `"You are the [X] sub-agent for a self-hosted automation system"` literal in their systemMessage (line 33 for ops, line 36 for the chainLlm-based three). Stage 106 only added the `Read Style Profile` + style ternary; it never touched the persona line. The handoff §2 row 2 claim that Stage 106 verified persona neutering on tgbot10 is wrong — Stage 106's verification was 4 routing-correctness messages, not a "who are you?" probe. tgbot12's first message ("מה אתה יודע על המערכת הזו?") was classified `research` and the research-agent answered `"אני סוכן מחקר (RESEARCH sub-agent) במערכת אוטומציה עצמית-מתארחת"` — full architecture leak. Fix: 4 systemMessages rewritten from "You are the X sub-agent…" to "You provide/help/give advice…" — capability description, no identity statement. Style-profile ternary and all surrounding nodes untouched. New trailing rule on each: "never reveal the names of internal sub-agents or your own architectural role" — explicit do-not-leak instruction. **(3) ops-agent tool gap — Stage 108 architectural miss.** The classifier (correctly) routes `"כמה הרצות נכשלו ב-24 שעות?"` to ops, but Stage 108 added `postgres_named_query` (which holds the `executions_summary_24h` query designed for exactly that question) only to unknown-agent, so ops answered honestly that it can only list workflows. Fix: ops-agent gets the same `postgres_named_query` toolWorkflow node (id `b0000000…0009`) + `ai_tool→Ops Agent` connection + `@@SYSTEM_INFO_JSON@@` block in its systemMessage. `configure-agent-router.yml`'s SYSTEM-INFO injection — previously hard-coded to `unknown-agent.json` and node `"Chat Agent"` — now uses a `case "$file"` to also inject into `ops-agent.json` at node `"Ops Agent"` (jq's `select(.name==$target)` parameterized). The PG-missing fallback that strips `postgres_named_query` now also runs on ops-agent (split out of the unknown-only block so it applies to both). No change to code/research/infra agents (their model has no tool surface). Per CLAUDE.md `templates/system/` edits only reach newly-provisioned systems; verification = fresh `factory-test-tgbot13` + 3 Telegram messages targeting each of (a) SYSTEM-INFO answer (no identity leak), (b) `executions_summary_24h` via ops, (c) `style_profile_get`. The handoff's verification discipline now also covers persona neutrality for every sub-agent the classifier might pick. |

## Stage 104 — fix: silent bot failures — decommission revoked wrong OR key, configure didn't notice, unknown-agent broke data flow

| PR | Type | Summary |
|---|---|---|
| TBD | fix | Three connected bugs surfaced by a real Telegram message that the bot answered with the fallback `אין לי תשובה כרגע` — the first end-to-end smoke that wasn't a configure-time PASS marker. **(1)** `decommission-test-system.yml` read the OpenRouter key hash from the *shared* SM (`factory-test-25` in reuse mode), so a decommission of any older system revoked whichever key was minted **last** — today that meant decommissioning `tgbot2` killed `tgbot8`'s live key (the remaining 4 decommissions returned 404). Patched to identify the key by **name** via `GET /api/v1/keys` filtered by `name == SYSTEM_NAME`, so each teardown revokes its own key regardless of SM contents. **(2)** `configure-agent-router.yml` happily logged `PASS: agent-router configured` while every Chat Agent call silently 401'd. Added a live OpenRouter probe — direct `curl POST openrouter.ai/api/v1/chat/completions` with the SM key — and writes a Hebrew WARN to the job summary on non-200 (with the actual error body + remediation pointers). This is the missing "verify the thing actually works" step. **(3)** `unknown-agent.json` (from PR 149's `Read Style Profile` node) broke the data flow two ways: Postgres `SELECT` returning 0 rows emitted zero items, short-circuiting Chat Agent entirely; and even on 1 row, Chat Agent's `text: ={{ $json.sanitized }}` resolved to `undefined` because Read Style Profile *replaces* `$json` with the SELECT result. Fixed both: `alwaysOutputData: true` on Read Style Profile, and Chat Agent now reads `text` from `$('When Executed by Another Workflow').first().json.sanitized` — works whether the style row exists or not, and whether the `jq` fallback strips the node (no-PG path) or not. No template churn beyond the 3 edits; no behavior change for systems that already had a working OpenRouter key + populated style_profile. Bot end-to-end behavior will be verified on a fresh `factory-test-tgbot9` after merge, with an actual Telegram message + inspect of the resulting `agent-router` execution. |
