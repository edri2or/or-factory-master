## Stage 113 — fix: postgres_named_query — drop specifyInputSchema, trigger back to passthrough (proven on tgbot15 with the operator sending a real message)

| PR | Type | Summary |
|---|---|---|
| TBD | fix | Stages 110/111/112 all chased the wrong path: with `executeWorkflowTrigger` in `workflowInputs` mode + the `toolWorkflow` node carrying any flavour of `specifyInputSchema` (`manual` JSON Schema with enum, or `fromJson` example), n8n 1.121's bridge between LLM args and the trigger's `$json` shipped `query_name: null` regardless of what the LLM provided. Verified deterministically by two diagnostic rounds on `factory-test-tgbot14` (DEBUG1, after Stage 111's `workflowInputs`) and on `factory-test-tgbot15` (DEBUG2, after Stage 112's `schemaType:"fromJson"`): both surfaced identical evidence — `Keys at root: ["query_name"]`, `typeof query_name: object` (JS quirk: `typeof null === "object"`), `_raw_input: {"query_name": null}`. The schema → Zod → tool-args → trigger-pickBy pipeline in toolWorkflow v2.1 simply does not propagate the LLM's structured value to the sub-workflow. **DEBUG3 ran with `specifyInputSchema:false` (drop schema entirely) + trigger back to `passthrough`** — tgbot15 then produced a real bot reply ("your style profile is still empty"), and `inspect_n8n_execution` confirmed execution 28 terminates at **`Format Output`** (the success path), not `Unknown Query`. That's the first end-to-end success for the tool. Fix in main: (1) `templates/system/workflows/n8n/postgres-named-queries.json` — trigger back to `inputSource:"passthrough"` (the `workflowInputs` config was load-bearing only for Stages 110-112's schema dance, now reverted), Normalize Code node updated to extract `query_name` from `$json.query` as a raw string against the 4-value whitelist (also tries JSON.parse for safety). (2) `unknown-agent.json` + `ops-agent.json` — `postgres_named_query` toolWorkflow nodes lose `specifyInputSchema`, `schemaType`, `jsonSchemaExample`. Tool description rewritten to be unambiguous about the call shape ("Call this tool with ONE input — a single string that is EXACTLY one of these 4 query names, do not wrap it in JSON or any object, just send the raw string"). Inline code-comment on the Normalize node records the 3-stage diagnostic so a future reader sees why this specific shape was chosen. Per CLAUDE.md `templates/system/` edits reach only newly-provisioned systems; tgbot15's diagnostic copy is already shaped this way (the success was on tgbot15 *with* the diagnostic Telegram node — that node lives in tgbot15's own repo copy and won't migrate to other systems). Verification on a fresh `factory-test-tgbot16` after merge: 3 real Telegram messages, all 3 should produce success-path executions on `postgres-named-queries` (failingNode/lastNode = `Format Output`) and zero `Unknown Query` terminations. |
