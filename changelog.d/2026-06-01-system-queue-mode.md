### queue mode אופציונלי למערכות (Redis + n8n worker) — מאחורי מתג `QUEUE_MODE`, ברירת-מחדל כבוי

- **שלב 1 — מתג + מעטפות SM (תשתית רדומה).** `provision-system.yml` מקבל 4 מעטפות SM חדשות ב-`RUNTIME_SHELLS` (`railway-redis-service-id`, `railway-redis-volume-id`, `railway-worker-service-id`, `redis-password`) ומגדיר משתנה-ריפו חדש `QUEUE_MODE=false` לכל מערכת חדשה. ה-deploy של המערכת (`templates/system/.github/workflows/deploy-railway-cloudflare.yml`) קורא את `vars.QUEUE_MODE` ומנרמל ל-`QM` בוליאני בראש שלב "Provision Railway" — עדיין בלי שום שירות חדש. כש-`QUEUE_MODE` כבוי (ברירת-המחדל), הפריסה זהה-בייט למצב הקודם: אין Redis, אין worker, ו-n8n מקבל בדיוק את אותם 15 משתני-סביבה. ה-golden (`tests/golden/system/MANIFEST.sha256`) רוענן לשינוי ה-deploy template.
- **שלב 2 — שירות Redis מותנה.** בלוק Redis נוסף ל-deploy template, עטוף `if [ "$QM" = "true" ]`, אחרי שירות n8n. כשהמתג דלוק הוא מקים שירות `redis:7-alpine` (בדפוס SM-first של Postgres: id מ-SM → find-by-name → `serviceCreate` → שמירת `railway-redis-service-id`), volume ל-AOF ב-`/data` (`railway-redis-volume-id`), מייצר `redis-password` אידמפוטנטית, ומכריח אימות+AOF דרך start command (`redis-server --requirepass "$REDIS_PASSWORD" --appendonly yes`, ב-`serviceInstanceUpdate`). `REDIS_FIRST_TIME` guard מונע redeploy מיותר ב-re-run. כשהמתג כבוי הבלוק כולו מדולג — אין Redis, פריסה זהה-בייט. golden רוענן.
- **שלב 3 — env ראשי מותנה + שירות worker מותנה.** משתני התור מתווספים ל-n8n הראשי דרך מיזוג `{...16 בסיס...} + $QUEUE_EXTRA`, כש-`QUEUE_EXTRA='{}'` כשהמתג כבוי → פלט זהה-בייט ל-upsert המקורי (הוכח: OLD == NEW). כשדלוק, ה-main מקבל `EXECUTIONS_MODE=queue`, `QUEUE_BULL_REDIS_HOST/PORT/PASSWORD`, `N8N_DEFAULT_BINARY_DATA_MODE=database`, `N8N_GRACEFUL_SHUTDOWN_TIMEOUT`, ובלעדית-main `N8N_DISABLE_PRODUCTION_MAIN_PROCESS=true` + `OFFLOAD_MANUAL_EXECUTIONS_TO_WORKERS=true`. נוסף שירות `worker` מותנה (אותו `$N8N_IMAGE`, start command `n8n worker`, env משותף: DB refs + encryption key + queue vars + binary mode + runners + graceful shutdown, בלי הדגלים הבלעדיים-main, בלי דומיין/Caddy), בדפוס SM-first + `WORKER_FIRST_TIME` guard, id נשמר ב-`railway-worker-service-id`. ייצור סיסמת Redis אוחד למקום אחד (לפני ה-env), והבלוק של Redis הוקדם בהתאם. כשהמתג כבוי — אין worker, פריסה זהה-בייט. golden רוענן.
- **שלב 4 — תיעוד.** `docs/roadmap.md` קיבל מקטע "Phase J — optional per-system queue mode" (המתג, ברירת-מחדל כבוי, עלות ~$10–20, גידול Postgres עקב binary ב-DB, הדלקה על מערכת קיימת, מה לא נגעו). `templates/system/AGENTS.md.template` קיבל מקטע "Queue mode (scaling)" + 4 מעטפות ה-SM החדשות סומנו *(queue mode only)* ברשימת ה-runtime shells + bullet ב-"What was provisioned". golden רוענן (hash של AGENTS.md + rendered/AGENTS.md).
- **שלב 4b — מתג דרך dispatch input (override).** התגלה שהמתג היה משתנה-ריפו בלבד, ולפקטורי אין כלי להגדיר משתנה-ריפו (וגם Or לא נוגע ב-UI) — כך שלא הייתה דרך hands-off להדליק queue mode. נוסף ל-`deploy-railway-cloudflare.yml` input אופציונלי `queue_mode` ל-`workflow_dispatch` שגובר על `vars.QUEUE_MODE` (input לא-ריק מנצח; אחרת נופלים למשתנה הדביק; ברירת-מחדל off). עכשיו הדלקה/כיבוי לריצה אחת = dispatch יחיד שהסוכן יכול לעשות, בלי לגעת במשתנה. נרמול ה-`QM` עודכן בהתאם, ואומת בטבלת-אמת (5 מקרים). תועד ב-roadmap (Phase J) וב-AGENTS template. off-path נשאר זהה-בייט. golden רוענן (deploy template + AGENTS.md).
