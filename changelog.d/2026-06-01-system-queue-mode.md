### queue mode אופציונלי למערכות (Redis + n8n worker) — מאחורי מתג `QUEUE_MODE`, ברירת-מחדל כבוי

- **שלב 1 — מתג + מעטפות SM (תשתית רדומה).** `provision-system.yml` מקבל 4 מעטפות SM חדשות ב-`RUNTIME_SHELLS` (`railway-redis-service-id`, `railway-redis-volume-id`, `railway-worker-service-id`, `redis-password`) ומגדיר משתנה-ריפו חדש `QUEUE_MODE=false` לכל מערכת חדשה. ה-deploy של המערכת (`templates/system/.github/workflows/deploy-railway-cloudflare.yml`) קורא את `vars.QUEUE_MODE` ומנרמל ל-`QM` בוליאני בראש שלב "Provision Railway" — עדיין בלי שום שירות חדש. כש-`QUEUE_MODE` כבוי (ברירת-המחדל), הפריסה זהה-בייט למצב הקודם: אין Redis, אין worker, ו-n8n מקבל בדיוק את אותם 15 משתני-סביבה. ה-golden (`tests/golden/system/MANIFEST.sha256`) רוענן לשינוי ה-deploy template.
- **שלב 2 — שירות Redis מותנה.** בלוק Redis נוסף ל-deploy template, עטוף `if [ "$QM" = "true" ]`, אחרי שירות n8n. כשהמתג דלוק הוא מקים שירות `redis:7-alpine` (בדפוס SM-first של Postgres: id מ-SM → find-by-name → `serviceCreate` → שמירת `railway-redis-service-id`), volume ל-AOF ב-`/data` (`railway-redis-volume-id`), מייצר `redis-password` אידמפוטנטית, ומכריח אימות+AOF דרך start command (`redis-server --requirepass "$REDIS_PASSWORD" --appendonly yes`, ב-`serviceInstanceUpdate`). `REDIS_FIRST_TIME` guard מונע redeploy מיותר ב-re-run. כשהמתג כבוי הבלוק כולו מדולג — אין Redis, פריסה זהה-בייט. golden רוענן.
- **שלב 3 — env ראשי מותנה + שירות worker מותנה.** משתני התור מתווספים ל-n8n הראשי דרך מיזוג `{...16 בסיס...} + $QUEUE_EXTRA`, כש-`QUEUE_EXTRA='{}'` כשהמתג כבוי → פלט זהה-בייט ל-upsert המקורי (הוכח: OLD == NEW). כשדלוק, ה-main מקבל `EXECUTIONS_MODE=queue`, `QUEUE_BULL_REDIS_HOST/PORT/PASSWORD`, `N8N_DEFAULT_BINARY_DATA_MODE=database`, `N8N_GRACEFUL_SHUTDOWN_TIMEOUT`, ובלעדית-main `N8N_DISABLE_PRODUCTION_MAIN_PROCESS=true` + `OFFLOAD_MANUAL_EXECUTIONS_TO_WORKERS=true`. נוסף שירות `worker` מותנה (אותו `$N8N_IMAGE`, start command `n8n worker`, env משותף: DB refs + encryption key + queue vars + binary mode + runners + graceful shutdown, בלי הדגלים הבלעדיים-main, בלי דומיין/Caddy), בדפוס SM-first + `WORKER_FIRST_TIME` guard, id נשמר ב-`railway-worker-service-id`. ייצור סיסמת Redis אוחד למקום אחד (לפני ה-env), והבלוק של Redis הוקדם בהתאם. כשהמתג כבוי — אין worker, פריסה זהה-בייט. golden רוענן.
- **שלב 4 — תיעוד.** `docs/roadmap.md` קיבל מקטע "Phase J — optional per-system queue mode" (המתג, ברירת-מחדל כבוי, עלות ~$10–20, גידול Postgres עקב binary ב-DB, הדלקה על מערכת קיימת, מה לא נגעו). `templates/system/AGENTS.md.template` קיבל מקטע "Queue mode (scaling)" + 4 מעטפות ה-SM החדשות סומנו *(queue mode only)* ברשימת ה-runtime shells + bullet ב-"What was provisioned". golden רוענן (hash של AGENTS.md + rendered/AGENTS.md).
- **שלב 4b — מתג דרך dispatch input (override).** התגלה שהמתג היה משתנה-ריפו בלבד, ולפקטורי אין כלי להגדיר משתנה-ריפו (וגם Or לא נוגע ב-UI) — כך שלא הייתה דרך hands-off להדליק queue mode. נוסף ל-`deploy-railway-cloudflare.yml` input אופציונלי `queue_mode` ל-`workflow_dispatch` שגובר על `vars.QUEUE_MODE` (input לא-ריק מנצח; אחרת נופלים למשתנה הדביק; ברירת-מחדל off). עכשיו הדלקה/כיבוי לריצה אחת = dispatch יחיד שהסוכן יכול לעשות, בלי לגעת במשתנה. נרמול ה-`QM` עודכן בהתאם, ואומת בטבלת-אמת (5 מקרים). תועד ב-roadmap (Phase J) וב-AGENTS template. off-path נשאר זהה-בייט. golden רוענן (deploy template + AGENTS.md).
- **שלב 5 (אימות חי — סבב תיקון).** מערכת-טסט `factory-test-qmode` הוקמה (reuse) ונפרסה עם `queue_mode=true`: 5 שירותים (Postgres/n8n/caddy/redis/worker) עלו SUCCESS, ה-worker התחבר לתור כ-`n8n worker`, וה-n8n הראשי קיבל את כל חוזה התור — **הליבה הוכחה חי**. האימות חשף שני באגים שתוקנו: (1) `N8N_DEFAULT_BINARY_DATA_MODE=database` אינו ערך חוקי ב-n8n 1.121 (`default|filesystem|s3`) — תוקן ל-`default` (מצב מגובה-DB, הכוונה המקורית) ב-env הראשי וב-worker; (2) ה-Redis עלה ללא סיסמה/AOF כי Railway לא מרחיב `"$VAR"` ב-start command והערך המצוטט נדחק — תוקן להזרקת הסיסמה ה**מילולית** (hex, ללא ציטוט/`$`) דרך `jq --arg`, ובאופן **self-healing אידמפוטנטי** (קוראים את ה-start command הנוכחי ומעדכנים רק אם שונה, כך ש-re-deploy מתקן Redis שגוי בלי redeploy מיותר). הוסר ה-`REDIS_FIRST_TIME` ועדכון ה-env המיותר של REDIS_PASSWORD על שירות ה-Redis. golden + תיעוד עודכנו.
- **שלב 5 (אימות חי — סבב תיקון #2: קריסת-סדר של Redis).** האימות החי על `factory-test-qmode2` חשף ש-n8n הראשי **קרס באמצע הפריסה**: ה-main קיבל את הגדרות התור והתחבר ל-Redis, ואז ה-Redis עבר redeploy (להחלת סיסמה+AOF) מתחת לרגליו → `connect ETIMEDOUT` → `Queue errored` → `Exiting process due to Redis connection error`; אחרי האתחול, ה-webhook של ה-notifier החזיר 404 וההודעה "n8n מוכן" מעולם לא נשלחה (כך Or זיהה את התקלה). **התיקון (סדר):** בלוק ה-Redis (כולל ייצור-סיסמה + start command + **המתנה ל-Redis בריא**, poll עד `SUCCESS`) הוקדם אל **לפני** ה-env של ה-n8n הראשי — כך ה-main מתחבר פעם אחת ל-Redis יציב ולא קורס. בנוסף הוסר `N8N_DISABLE_PRODUCTION_MAIN_PROCESS=true` (דגל-כוונון אופציונלי ולא-מאומת; בלעדיו ה-main נשאר fallback executor עמיד יותר למסירת webhooks). `OFFLOAD_MANUAL_EXECUTIONS_TO_WORKERS` נשמר (n8n עצמו ממליץ). off-path נשאר זהה-בייט; golden + roadmap עודכנו. נותר: אימות חי על מערכת טרייה (`factory-test-qmode3`) שאין `Queue errored`/קריסה וההודעה נשלחת.
- **שלב 5 (אימות חי — סבב תיקון #3: ה-worker רץ כ-main).** אחרי תיקון-הסדר, האימות על `factory-test-qmode3` הראה ש**אין יותר קריסה** (אין `Queue errored`/`Exiting`) — אבל ה-webhooks הרשומים נתקעו ב-timeout. החקירה: ה-main **כן** מתזמר ל-Redis (`Enqueued execution 1/2/3`), אבל שירות ה-"worker" עלה ב**מצב main** (`n8n ready on port 5678` / `Editor is now accessible` במקום `n8n worker is now ready`) ולכן אף אחד לא צרך את התור. הסיבה: ה-entrypoint של image של n8n מריץ `exec n8n "$@"`, כך ש-start command `n8n worker` הפך ל-`n8n n8n worker` (לא חוקי → נפילה ל-main). **התיקון:** start command של ה-worker שונה מ-`n8n worker` ל-`worker` (ה-entrypoint מוסיף `n8n`, ומקבלים `n8n worker` תקין). לא באג-גרסה ולא #17395 — תיקון מילה אחת. off-path זהה-בייט; golden+roadmap+AGENTS עודכנו. נותר: אימות חי על מערכת טרייה (`factory-test-qmode4`) שה-worker צורך jobs וה-webhook נענה + ההודעה נשלחת.
- **שלב 5 (אימות חי — סבב תיקון #4: ה-worker start command לא הוחל).** ב-qmode4 התברר ש**גם** `worker` העלה את ה-worker כ-main, ועם פריסה אחת בלבד — כלומר ה-start command **בכלל לא הוחל** (אילו `worker` היה מוחל כ-exec override הקונטיינר היה קורס על "worker: not found"; במקום זה רץ ברירת-המחדל של ה-image = main). מחקר: ה-start command של Railway על שירות-image **דורס את ה-ENTRYPOINT ב-exec form**, אז הערך הנכון הוא **`n8n worker`** (תבניות n8n הרשמיות של Railway: `n8n worker --concurrency=10`). הסיבה שזה לא הוחל: הוגדר בתוך ה-`WORKER_FIRST_TIME` gate מיד אחרי `serviceCreate` (תוך כדי הבילד הראשוני) ולא טריגר redeploy מחיל — בעוד ש-Redis (compale-then-set לא-מגודר, אחרי השהיה) כן הוחל. **התיקון:** הערך הוחזר ל-`n8n worker`, וה-start command מוגדר עכשיו ב-**compare-then-set לא-מגודר** (כמו Redis) **אחרי** ה-env upsert, ועם **המתנה לפריסת ה-worker עד `SUCCESS`** — כך ה-redeploy שמעביר את ה-worker למצב-worker אכן נוחת. ה-env נשאר `WORKER_FIRST_TIME`-gated. off-path זהה-בייט; golden רוענן. נותר: אימות חי על מערכת טרייה (`factory-test-qmode5`) שלוג ה-worker = `n8n worker is now ready`, ה-jobs נצרכים, ה-webhook נענה, וההודעה נשלחת.
- **שלב 5 (אימות חי — סבב תיקון #5: ה-worker נבנה מ-Dockerfile, דטרמיניסטי).** ב-qmode5 התברר ש**גם** `n8n worker` (שהוחל הפעם — נראה redeploy) העלה את ה-worker כ-main. מסקנה חד-משמעית: ה-start command של Railway על שירות-image **מתווסף כ-CMD** ל-entrypoint (`exec n8n "$@"`), אז `n8n worker` → `n8n n8n worker` → נפילה ל-main; `worker` הוא הערך הנכון אך ההחלה שלו הוכחה שברירית-תזמון. **התיקון הסופי (דטרמיניסטי):** ה-worker נבנה מ-`templates/system/Dockerfile.worker` (`FROM n8nio/n8n:1.121.0` + `CMD ["worker"]`) דרך `RAILWAY_DOCKERFILE_PATH`, בדיוק כמו Caddy — ה-image מריץ `n8n worker` מעצמו, בלי תלות ב-start command של Railway. שירות ה-worker שונה מ-`source: {image}` ל-`source: {repo}`, נוסף `RAILWAY_DOCKERFILE_PATH: "Dockerfile.worker"` ל-env, ובלוק ה-start-command הוסר (נשארה ההמתנה לפריסה). הערה: יש לשמור את גרסת n8n ב-Dockerfile.worker מסונכרנת עם `N8N_IMAGE`. off-path זהה-בייט (הבלוק QM-gated); golden רוענן (Dockerfile.worker + deploy template). נותר: אימות חי על `factory-test-qmode6` שלוג ה-worker = `n8n worker is now ready`, ה-jobs נצרכים, וההודעה נשלחת.
