## feat: telegram-bot-fuzzy-resolver — file catalog layer (Stage 1)

| Type | Summary |
|---|---|
| feat | Stage 1 of telegram-bot-fuzzy-resolver: the data layer for a deterministic file resolver that lets every provisioned system's Telegram bot map an imprecise/misspelled file name to a real repo path (instead of giving up, demanding an exact path, or inventing one). New `templates/system/workflows/n8n/file-catalog-refresh.json` — an hourly n8n workflow (`scheduleTrigger 0 * * * *`) that reuses `github-readonly.json`'s GitHub-App JWT mint+cache chain (opaque token, `onError: continueRegularOutput`), fetches the system repo's full file list in one call (`GET /repos/edri2or/@@SYSTEM_NAME@@/git/trees/HEAD?recursive=1`), extracts every `type:"blob"` path, and upserts the array into Postgres `file_catalog` keyed by `@@CHAT_ID@@` (mirrors `style-refresh.json`'s upsert). Soft-fail throughout: a `Has Paths?` guard skips the upsert on an empty/failed fetch so an existing catalog is never wiped, and a `truncated` flag is carried if GitHub caps the tree (>100k files / 7MB — not expected for factory systems). `db-setup.json` gains `CREATE TABLE IF NOT EXISTS file_catalog (chat_id BIGINT PRIMARY KEY, paths JSONB NOT NULL, refreshed_at TIMESTAMPTZ NOT NULL DEFAULT now())` and `'file_catalog'` in its verification SELECT. `configure-agent-router.yml` registers + activates the workflow (gated on the GitHub-App JWT credential + Postgres + chat id; same `_upsert_wf` pattern as `style-refresh`) and triggers one initial `/rest/.../run` so the catalog is seeded at install rather than waiting for the first cron. Only `@@…@@` placeholders, no secret literals. Template-only; no live system touched. Verified locally: all n8n JSON valid, `yamllint` + `actionlint -shellcheck "shellcheck -S error"` clean on the workflow, `validate-templates.sh` + BATS green. |

## feat: telegram-bot-fuzzy-resolver — deterministic resolver in the router (Stage 3)

| Type | Summary |
|---|---|
| feat | Stage 3 (core) of telegram-bot-fuzzy-resolver, in `templates/system/workflows/n8n/agent-router.json`. The `Classify Intent` prompt now also emits `entity_mention` (the file name/path the user referenced, extracted verbatim — never corrected/invented; `intent`/`confidence`/single-line-JSON instruction preserved so the `eval_router.py --check` precheck stays green), and `Build Dispatch` parses + carries it. Three new nodes sit between `Build Dispatch` and `Route by Intent`: `Load Catalog` (Postgres `SELECT paths FROM file_catalog WHERE chat_id=…`, `onError:continueRegularOutput` + `alwaysOutputData`), `Resolve Entity` (a ~zero-dependency Jaro-Winkler matcher scoring the mention against every catalog path's basename AND full path, case-insensitive), and a `Resolver Gate` switch. Thresholds (research §7.3): **≥0.85** → inject the verified path into `sanitized` (`[המערכת אימתה: הנתיב … read_file:<path>]`) and continue to the normal agent; **0.70–0.84** → STOP and reply (single candidate → "התכוונת ל-X? (כן/לא)"; multiple → a numbered 2–4 list, "ענה במספר."); **<0.70** → STOP and offer a folder listing; **no mention / empty catalog / any error** → continue unchanged (graceful degrade — the resolver never blocks normal conversation, and a missing Postgres just falls back to today's behavior). STOP replies flow straight into `Egress Validation`, so the existing egress policy still runs. `configure-agent-router.yml`'s router install step now also substitutes `@@CRED_POSTGRES_ID@@` + `@@CHAT_ID@@`. No secret literals. Template-only. Verified locally: JSON valid (15 nodes), `eval_router.py --check` green (battery 250 + prompt intact), `yamllint` + `actionlint/shellcheck` clean, and a Node unit-test of the matcher (`cnfig.json`→`config.json` 0.97, `agnts.md`→`AGENTS.md` 0.97, `app.ts`→`src/app.ts` 1.0, unmatched→folder-offer, no-mention/empty-catalog→skip). |

(1) The long-standing wrong example `read_file:AGENT.md` (a file that does not exist) is corrected to `read_file:AGENTS.md` everywhere it appeared — both the `systemMessage` and the `github_readonly` toolDescription of both agents (3 occurrences in ops-agent, 5 in unknown-agent, incl. bare `AGENT.md` mentions). Read-only; no new secret; `html_url` links still allowed. Template-only. Verified locally: zero `read_file:AGENT.md` left, all n8n JSON valid, `validate-templates.sh` green. |
