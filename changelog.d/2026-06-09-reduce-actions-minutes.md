## fix: reduce-actions-minutes — cheap Actions-minute safety ceilings (Stage 1/3)

| Type | Summary |
|---|---|
| fix | Stage 1 of the reduce-actions-minutes development, opened after an investigation traced a surprise GitHub charge (677 ₪ ≈ $223.55) to the GitHub receipt's **$215.55 of GitHub Actions usage for May 2026** (+ $8 Team plan) — the 3,000 included minutes were exhausted by May 6, driven by the May consolidation sprint's ~70 throwaway `factory-test-*` provision+deploy cycles (several redeployed many times: `qmode` ×7, `tgbot` ×16). Adds three behavior-preserving guardrails so a heavy sprint can't silently blow the quota again: (1) explicit `timeout-minutes` ceilings on the two heaviest jobs — `provision-system.yml` job `provision` (30) and the system template `deploy-railway-cloudflare.yml` job `deploy` (45) — replacing GitHub's 6-hour default so a stuck run can't burn hours; (2) `concurrency: { group: ${{ github.workflow }}-${{ github.ref }}, cancel-in-progress: true }` on all nine per-PR CI gates (the 5 factory + 4 system-template `changelog-check`/`pipeline-tests`/`secret-scan`/`supply-chain-check`/`playground-tests`) so a new push cancels its own superseded run instead of letting both finish; (3) widened the two 6-hourly audits (`factory-health-audit`, `system-runtime-audit`) to daily — downtime is already caught sub-minute by Better Stack + the ~hourly `bs-incidents-to-telegram` relay, so the 6h cadence was a redundant heartbeat. System golden refreshed (`tests/golden/system/MANIFEST.sha256`) for the 5 template-workflow edits; no envsubst variables changed so the allow-list stays byte-identical. Behavioral proof (a live throwaway test-system provision+deploy inside the new ceilings) is Stage 3. An independent GitHub Actions spending limit was set by Or in parallel. |

## fix: prove-on-test-system — request workflows:write so it can prove changes to .github/workflows/* files

| Type | Summary |
|---|---|
| fix | Discovered while live-proving the Stage 2 deploy change: `prove-on-test-system.yml` minted its per-system App token with only `contents`/`pull_requests`/`actions` write, so pushing a change to a `.github/workflows/*` file was rejected by GitHub (`refusing to allow a GitHub App to create or update workflow ... without 'workflows' permission`). The tool's original use case was syncing n8n workflow JSONs (not `.github/workflows`), so this gap only surfaced when proving a change to the deploy workflow itself. Added `"workflows":"write"` to the token-permission request — the per-system App already holds `workflows:write` on its single repo (granted by `register-system-app.yml`), so this is a scope-down request it already satisfies; blast radius stays one throwaway test repo. Without this, the "prove → merge" loop could not validate any change to a system's own deploy/CI workflows before merge. |

| fix | Stage 2 of the reduce-actions-minutes development — the cost-optimization half (the biggest deploy-minute saving), in the system template `deploy-railway-cloudflare.yml`. A new early step **`Detect throwaway test system (skip HMAC edge)`** computes `SKIP_EDGE=true` when `vars.GCP_PROJECT_ID` matches the reuse/test prefix `^(factory-test-|v2-test-|or-test-)` — the exact namespace `provision-system.yml` validates `shared_gcp_project` against (a real system's `GCP_PROJECT_ID` is its `SYSTEM_NAME` or an adopted id, and adopt mode hard-refuses `factory-test-25`, so **real deploys stay byte-identical**) — and exports it via `$GITHUB_ENV`. When set, five edge steps are `if:`-skipped on throwaway test deploys (`Generate or read webhook HMAC secret`, `Provision Caddy gateway service on Railway`, `Caddy gateway smoke tests`, `Migrate public domain to Caddy`, `Verify gateway end-to-end + summary`), and the queue-mode n8n-worker Docker build is skipped too (`QM=true && SKIP_EDGE!=true`). Because the Caddy steps never run, the `caddy-railway-service-id` secret stays absent, `MIGRATED_TO_CADDY` stays false, n8n keeps its public domain, and `/healthz` is served by n8n directly — preserving the contract `system-runtime-audit.yml` + the Stage-3 probe depend on. This removes the Caddy Docker build, its ~10-min `/health` poll, and the domain-swap's ~10-min cert poll from every test deploy. Independently, two **readiness** wait-loops were tightened for ALL systems (redis-ready 18→9 cycles / ~3min→~90s; n8n-owner-ready 36→24 / ~6min→~4min) and the two Caddy webhook-warmup loops 18→12 — while **every cert/Let's-Encrypt loop and both Docker-build readiness loops were left at their generous ceilings** so no legitimate slow issuance can false-fail. System golden refreshed (only `deploy-railway-cloudflare.yml`'s hash changed); yamllint + shellcheck(bash) clean. Behavioral proof — a live throwaway test deploy showing the 5 edge steps skipped, `/healthz` 200 on n8n's domain, and the minute drop, plus a real-path `SKIP_EDGE=false` assertion — is Stage 3. |
