## feat: ops-agent-live-telemetry — github-readonly sub-workflow + in-n8n token mint (Stage 2)

| Type | Summary |
|---|---|
| feat | Stage 2 of live read-only telemetry for the ops-agent. New `templates/system/workflows/n8n/github-readonly.json` — a `toolWorkflow` sub-workflow (`factory-master: github-readonly`) on the same skeleton, with a GitHub-App installation-token mint stage shared by all branches. `Normalize Input` resolves the raw string to `ci_runs` \| `recent_commits` \| `open_prs`. Token stage: `Token Cache Check` reads `$getStaticData('global').github_token_cache` and an `IF` (`Token Valid?`) skips minting while the token has >300s left; otherwise `Build JWT Claims` (iat−60s, exp+540s, iss=`@@GITHUB_APP_ID@@`) → built-in **JWT node** (`n8n-nodes-base.jwt`, sign, RS256 via a `jwtAuth` PEM credential `@@CRED_GITHUB_JWT_ID@@` — so the container needs no `NODE_FUNCTION_ALLOW_BUILTIN=crypto`) → POST `/app/installations/@@GITHUB_INSTALLATION_ID@@/access_tokens` → `Cache Token` persists `{token,exp}` in static data. Each branch then GETs the matching REST endpoint (`actions/runs` / `commits` / `pulls?state=open`, `per_page=10`) for `edri2or/@@SYSTEM_NAME@@` with a dynamic `Authorization: Bearer` header (token is runtime, never stored). `Format Output` → `{ ok, command, data }`, keeping `html_url` intact; a GitHub error envelope flips `ok:false`. The private key never enters the JSON (credential id only); the token is treated as opaque — no length/format regex anywhere. Read-only throughout (`onError: continueRegularOutput`). Verified locally: `jq .` valid, all five Code nodes pass `node --check`, no key material / token-length assumption present. JWT-node param names and REST array-vs-object shape are flagged for the live test-system verification. Template-only. |
