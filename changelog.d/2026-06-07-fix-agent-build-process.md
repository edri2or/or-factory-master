## process: dev-stage now requires a functional proof per stage (not just CI-green)

| Type | Summary |
|---|---|
| change | Redefined "verify each step" in `/dev-stage` (`.claude/commands/dev-stage.md`) after a pilot ("email-form agent") exposed the root flaw: every stage was coded and all real proof was deferred to one final "big-bang" stage (each stage literally wrote `☐ proof (live) — in stage 6`), so "verify each step" had collapsed into "CI is green". Step 3 now splits into **(a.1)** an *observable functional proof in the same stage* — feed the brick a real/pinned input and show the output is correct, pinned as a repeatable fixture, preferably checked MCP-independently (n8n Public API / a verification workflow) — and **(a.2)** the Playground CI check, which is stated as *necessary but not sufficient*. A stage may close without a functional proof only if it is genuinely non-runnable ("content only"). |
| change | `/dev-stage` Step 2 now mandates **bottom-up stage ordering** and forbids a plan that defers proof to a single late stage (the documented "big-bang integration" anti-pattern); the live/external endpoint (orchestrator/email/webhook) is the last brick to wire, never the first test. New Safety Rules 7–8 enforce both: no stage closes without its functional proof + green CI, and `☐ proof in stage N` is banned. |
| change | `templates/devplan/DEVPLAN.template.md` gains a mandatory per-stage **"הוכחה תפקודית (באותו שלב)"** field (real-input fixture, expected output, how it was observed) plus a discipline note; both shipped to every provisioned system. Shared-command mirror + system golden refreshed accordingly. |
| docs | Added the factory's agent-build documentation layer (based on the or-tok originals, generalized): `docs/research/agent-role-decomposition-planning.md` (role decomposition + the bottom-up "4 phases / 3 ordered gates" discipline that replaces a single end-of-line E2E gate; names the big-bang anti-pattern, fully sourced), `templates/agent-design-spec.md` (now carries a per-component **fixture → expected-output → test-method** column and a bottom-up gate list), and `docs/agent-isolation-testing.md` (the practical "prove a brick alone" guide: n8n Pin data, sub-workflow isolation, polling-trigger capture, and an **MCP-independent** result check via the n8n Public API / a dedicated verification workflow, with a runnable worked example). The full `build-agent` factory-port that consumes these is a deliberate follow-up. |
