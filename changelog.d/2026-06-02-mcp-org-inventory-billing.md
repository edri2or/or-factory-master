## feat: mcp-org-inventory-billing — org-wide cost & resource visibility tools

| Type | Summary |
|---|---|
| feat | Two new read-only MCP tools give the agent eyes on spend, so "how much am I spending and on what?" is answerable in any session without the operator opening the Cloud console. `inspect_all_org_resources` (`services/mcp-server/src/tools.ts`) calls Cloud Asset `searchAllResources` over the org scope (`organizations/905978345393`) and returns counts summarized by project, by asset type, and by project×type — "what exists everywhere" in one call (counts only, not per-resource rows; paginated with a 20-page cap, `truncated=true` if hit). `get_billing_costs` queries the Detailed-usage Billing→BigQuery export (`billing_export` dataset in `or-factory-master-control`) via a synchronous, parameterized `jobs.query`, grouping `SUM(cost)` by project and/or service over a lookback window (params `days` default 30, `groupBy` project\|service\|both). Both helpers live in `services/mcp-server/src/gcp-client.ts` (`searchAllOrgResources`, `queryBillingCosts`), reuse the existing `gcpFetch`/`gcpFetchPost` REST + bearer-token path (no `@google-cloud/*` libraries, no new npm deps), and run as the live Cloud Run runtime SA (the broker SA). |
| chore | Stage 3 closed (2026-06-02): both tools verified live in production after `deploy-mcp-server.yml` shipped the new build. `inspect_all_org_resources` returned an org-wide inventory (10,000 resources across 154 projects, top types: `secretmanager.googleapis.com/SecretVersion`=3,672, `Secret`=2,808, `serviceusage.googleapis.com/Service`=1,796; `truncated:true` at the 20-page cap). `get_billing_costs(days=30, groupBy=both)` returned the expected `available:false` warm-up response — the Detailed-usage Billing→BigQuery export was enabled the same day, first data lands within ~24h. Devplan `devplans/mcp-org-inventory-billing.md` flipped to `status: completed`, releasing the devplan CI gate. |
| chore | Prereqs done out-of-band (broker SA can't self-grant IAM): granted the broker SA `roles/billing.viewer` on billing account `014D0F-AC8E0F-5A7EE7`, `roles/cloudasset.viewer` at the org scope, and `roles/bigquery.dataViewer` + `roles/bigquery.jobUser` on `or-factory-master-control`; enabled `cloudasset`/`cloudbilling`/`bigquery` APIs; created the `billing_export` dataset (US multi-region) and enabled the Detailed-usage Billing→BigQuery export. Two robustness details: the BigQuery query carries `location:"US"` (the dataset is US multi-region — the job won't resolve the dataset otherwise), and a missing export table (it warms up ~24h after enablement) degrades to `available:false` with a friendly note instead of erroring. Permission gaps surface as a clear `permission_denied` result naming the exact role, not a crash. The `dispatch_workflow` allowlist is unchanged — these are read tools. |
