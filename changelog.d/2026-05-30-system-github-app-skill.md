## feat: every new system is self-describing about its GitHub App + can actually use it

| Type | Summary |
|---|---|
| feat | Closed the double gap where a newly provisioned system saw its dedicated GitHub App's three secrets (`github-app-id`, `github-app-private-key`, `github-app-installation-id`) without any explanation or a way to use them. **Knowledge:** `templates/system/AGENTS.md.template` now lists the three App secrets under the Secrets section and carries a short "GitHub App" section (what the App is, locked to this repo, the 5 permissions, that `APP_ID`/`APP_INSTALLATION_ID` are also repo vars, and the inside-a-workflow-vs-interactive rule), plus a new Forbidden Actions line forbidding printing/logging/disk-writing the private key or any minted token and forbidding fetching the key outside a workflow — all hardcoded text, so the `provision-system.yml` envsubst allow-list is untouched. **Capability:** new on-demand skill `templates/system/.claude/skills/github-app-operations/` (SKILL.md + `scripts/generate-app-token.sh`) ships into every future system via the existing `.claude` scaffold copy. The bundled script is the factory's `generate-app-token.sh` hardened so the private key is fed to openssl via process substitution from the `PRIVATE_KEY` env var (never written to disk; positional args become `<app-id> <install-id> [scope...]`). The skill documents two paths: PRIMARY (inside a GitHub Actions workflow — WIF auth, then `actions/create-github-app-token` for auto-mask+revoke, or the bundled script in runner memory; the only path that may touch the key) and INTERACTIVE (never fetch the key — use existing GitHub MCP/`gh` tools or dispatch a workflow), reflecting that the system has no interactive GCP creds and the gcp-hands broker returns results in issue comments. Also added a clarifying NOTE to the two creation commands (`github-app-bootstrap.md`, `create-github-app.md`) that the App already exists and to use the new skill instead. Template-only change — propagates to systems provisioned after this merge, not retroactively. |
