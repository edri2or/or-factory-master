### Fixed
- `templates/system/.github/workflows/deploy-railway-cloudflare.yml`: Caddy env-var upsert is now self-healing — re-asserts `HMAC_SECRET`, `N8N_TELEGRAM_WEBHOOK_SECRET`, `PORT`, `RATE_LIMIT_BURST`, `RATE_LIMIT_WINDOW`, `RAILWAY_DOCKERFILE_PATH` whenever any of them is missing or empty on the Railway service, instead of only on first creation. Idempotent: when all required vars are present the upsert is skipped (no needless Caddy redeploy, no Railway scheduler throttle). Root cause this fixes: a Railway-side auto-restart wiped the Caddy variable collection on `or-adhd-agent`, leaving the Caddyfile's `{$HMAC_SECRET}` / `{$N8N_TELEGRAM_WEBHOOK_SECRET}` path matchers empty so every request (including `/healthz` and the Telegram webhook) was rejected at the edge with 403, silencing the bot. With this change, re-running the deploy on a drifted system restores the vars; on a healthy system it stays a no-op.
