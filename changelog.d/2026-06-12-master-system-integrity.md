## fix: register‚Üíconfigure ordering preflight ‚Äî App registered but secrets missing is now loud (master-system-integrity Stage 2, B3)

| Type | Summary |
|---|---|
| fix | The system `configure-agent-router.yml` reads `github-app-{id,installation-id,private-key}` from Secret Manager and installs the `github_readonly` tool only when all three are present. Until now two very different states collapsed to the **same silent warning**: (a) the App is simply **not registered yet** (legitimate ‚Äî `register-system-app.yml` is a separate manual step), and (b) the App **was registered but its `github-app-*` secrets never reached SM** ‚Äî a real ordering break that silently strips a live capability with the job staying green. **Fix (single file, `templates/system/.github/workflows/configure-agent-router.yml`; additive):** expose `APP_ID_VAR=${{ vars.APP_ID }}` (the repo variable `register-system-app.yml` writes once the App is registered) to the configure step, and add a preflight in the github-app block ‚Äî `APP_ID` set **and** any `github-app-*` secret missing ‚Üí `GH_APP_BROKEN`. That feeds a **critical** row into the existing B4 damage report (STEP_SUMMARY ü©∫ + one `emit-event.sh` Telegram alert + `exit 1`), while `APP_ID` empty stays a green **warning** (a clean pre-registration birth is never falsely RED'd). No early exit ‚Äî configure still wires router/telegram/mcp (all independent of `github_readonly`), then the damage report exits 1 at the end; the existing B4 warning row is guarded (`&& [ -z "$GH_APP_BROKEN" ]`) so a broken state shows exactly one critical row, not a duplicate. **Proof:** `scripts/tests/github-app-preflight-test.sh` (new, in the Playground tests job, 5/5) ‚Äî registered+secrets-missing ‚Üí RED, registered+partial-secrets ‚Üí RED, not-registered ‚Üí green warning, healthy ‚Üí no row, broken suppresses the warning row. ## fix: HITL approval workflow published so the bot can actually send the ‚úÖ/‚ùå card (master-system-integrity Stage 3, B8)

| Type | Summary |
|---|---|
| fix | Stage 3's live bot test surfaced that the HITL write-action path was dead: asking the bot to run a workflow never produced the Telegram ‚úÖ/‚ùå approval card. **Root cause (found from the live `list_n8n_workflows` + the configure source):** `configure-agent-router.yml` upserts the `request-write-action` sub-workflow with `activate=no` (live `active=false`), while its **sibling production-invoked tool sub-workflows** (`github-readonly`, `railway-readonly`, `postgres-named-queries`) are all `activate=yes` (live `active=true`) and work. On n8n 2.x a `toolWorkflow` call from ops/unknown-agent loads only the **published** activeVersion, so the unpublished `request-write-action` made the `request_write_action` tool a silent no-op. The `no` was a stale pre-2.x "subworkflows stay inactive" assumption ‚Äî contradicted by the 2.x publish-insight already applied to its siblings. **Fix (single line + comment, `configure-agent-router.yml`):** publish it with `activate=yes` like the others; `executeWorkflowTrigger.trigger()` is a no-op so a published sub-workflow never self-fires (safe). Golden refreshed. Template-only ‚Äî reaches systems provisioned after merge; or-edri-4 gets it live via `refresh-system-agents.yml` + configure, and the definition of done is a real live ‚úÖ/‚ùå card on Telegram (Stage 3 re-test). |
