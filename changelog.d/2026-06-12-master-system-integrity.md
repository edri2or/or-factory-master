## fix: register→configure ordering preflight — App registered but secrets missing is now loud (master-system-integrity Stage 2, B3)

| Type | Summary |
|---|---|
| fix | The system `configure-agent-router.yml` reads `github-app-{id,installation-id,private-key}` from Secret Manager and installs the `github_readonly` tool only when all three are present. Until now two very different states collapsed to the **same silent warning**: (a) the App is simply **not registered yet** (legitimate — `register-system-app.yml` is a separate manual step), and (b) the App **was registered but its `github-app-*` secrets never reached SM** — a real ordering break that silently strips a live capability with the job staying green. **Fix (single file, `templates/system/.github/workflows/configure-agent-router.yml`; additive):** expose `APP_ID_VAR=${{ vars.APP_ID }}` (the repo variable `register-system-app.yml` writes once the App is registered) to the configure step, and add a preflight in the github-app block — `APP_ID` set **and** any `github-app-*` secret missing → `GH_APP_BROKEN`. That feeds a **critical** row into the existing B4 damage report (STEP_SUMMARY 🩺 + one `emit-event.sh` Telegram alert + `exit 1`), while `APP_ID` empty stays a green **warning** (a clean pre-registration birth is never falsely RED'd). No early exit — configure still wires router/telegram/mcp (all independent of `github_readonly`), then the damage report exits 1 at the end; the existing B4 warning row is guarded (`&& [ -z "$GH_APP_BROKEN" ]`) so a broken state shows exactly one critical row, not a duplicate. **Proof:** `scripts/tests/github-app-preflight-test.sh` (new, in the Playground tests job, 5/5) — registered+secrets-missing → RED, registered+partial-secrets → RED, not-registered → green warning, healthy → no row, broken suppresses the warning row. Golden refreshed (`scripts/check-system-golden.sh --update`). Template-only — reaches systems provisioned after merge; or-edri-4 gets it live via `refresh-system-agents.yml` + configure (the healthy path stays green; the broken `configure-before-register` path is proven deliberately at Stage 4 Day-0). |

## fix: HITL approval workflow published so the bot can actually send the ✅/❌ card (master-system-integrity Stage 3, B8)

| Type | Summary |
|---|---|
| fix | Stage 3's live bot test surfaced that the HITL write-action path was dead: asking the bot to run a workflow never produced the Telegram ✅/❌ approval card. **Root cause (found from the live `list_n8n_workflows` + the configure source):** `configure-agent-router.yml` upserts the `request-write-action` sub-workflow with `activate=no` (live `active=false`), while its **sibling production-invoked tool sub-workflows** (`github-readonly`, `railway-readonly`, `postgres-named-queries`) are all `activate=yes` (live `active=true`) and work. On n8n 2.x a `toolWorkflow` call from ops/unknown-agent loads only the **published** activeVersion, so the unpublished `request-write-action` made the `request_write_action` tool a silent no-op. The `no` was a stale pre-2.x "subworkflows stay inactive" assumption — contradicted by the 2.x publish-insight already applied to its siblings. **Fix (single line + comment, `configure-agent-router.yml`):** publish it with `activate=yes` like the others; `executeWorkflowTrigger.trigger()` is a no-op so a published sub-workflow never self-fires (safe). Golden refreshed. Template-only — reaches systems provisioned after merge; or-edri-4 gets it live via `refresh-system-agents.yml` + configure, and the definition of done is a real live ✅/❌ card on Telegram (Stage 3 re-test). |

## fix: tg-vision sends the real image bytes, not a filesystem reference (master-system-integrity Stage 3, B9)

| Type | Summary |
|---|---|
| fix | **Live root cause, proven from or-edri-4 execution 262 (not a guess):** the bot replied "couldn't analyze the image" for every real photo. Pulling the live execution data showed the data-URI it sent the vision model was `data:image/jpeg;base64,filesystem-v2...` — i.e. the base64 payload was a **storage reference**, not the image bytes. In `tg-vision`'s `To Base64 Data-URI` node, `let b64 = item.binary[propName].data || ''` reads `.data`, which in **filesystem binary mode** (`N8N_*_BINARY_DATA_MODE=filesystem`) is a reference string like `filesystem-v2:...` (truthy), so the correct `getBinaryDataBuffer` fallback was skipped and the garbage reference was base64-prefixed and sent. The vision model silently returned empty (no error) → the workflow's graceful fallback message. The MIME was correct (`image/jpeg`) — a researched octet-stream hypothesis was disproven by the live data. **Fix (`templates/system/workflows/n8n/tg-vision.json`, single node):** always `getBinaryDataBuffer(0, propName)` to read the real bytes (works in both in-memory and filesystem modes); drop the `.data` shortcut. Golden refreshed. Template-only — reaches new systems on merge; or-edri-4 gets it live via `refresh-system-agents.yml`. Definition of done: Or sends a photo and the bot describes it. |

## fix: ops-agent n8n API tool trims its 769 KB response so the agent stops 400-ing (master-system-integrity Stage 3, B10)

| Type | Summary |
|---|---|
| fix | **Live root cause, proven from or-edri-4 execution 284 (not a guess):** a text message that needed the workflow list (e.g. "deactivate workflow X") made the Ops Agent reply "I have no answer". Pulling the live execution data showed the model's **first** call succeeded (31,628 tokens, it called the `n8n API` tool), the `n8n API` tool returned **769,125 chars** (`GET /api/v1/workflows` returns every workflow's full `nodes`), and the **second** model call — carrying that 769 KB — failed with `400 "Provider returned error"` (context window exceeded), crashing the agent before it could send the HITL approval card. **Researched fix (sourced):** n8n's built-in HTTP Request Tool "Optimize Response" feature, which exists exactly to "reduce the amount of data passed to the LLM" ([n8n docs](https://docs.n8n.io/integrations/builtin/cluster-nodes/sub-nodes/n8n-nodes-langchain.toolhttprequest/); community [token-limit](https://community.n8n.io/t/context-token-limit-exceeded-in-ai-agent-chatbot-workflow/74968) / [optimize](https://community.n8n.io/t/how-to-optimize-ai-agent-token-sized-requests-to-stay-under-rate-limits/127057)). Exact parameter names taken from the n8n source (`ToolHttpRequest/descriptions.ts`). **Fix (`templates/system/workflows/n8n/ops-agent.json`, the `n8n API` tool node):** `optimizeResponse:true` + `responseType:"json"` + `dataField:"data"` + `fieldsToInclude:"selected"` + `fields:"id, name, active"` — the tool now returns only id/name/active per workflow (a few hundred bytes instead of 769 KB), and the toolDescription says so. Golden refreshed. Template-only — reaches new systems on merge; or-edri-4 gets it live via `refresh-system-agents.yml`. Definition of done: Or asks the bot to toggle a workflow and gets the ✅/❌ card. |
