## feat: email-form-intake — Gmail→form→Telegram proposal workflow shipped into every system (Stage 1/4)

| Type | Summary |
|---|---|
| feat | Stage 1 of the email-form-intake development. New `templates/system/workflows/n8n/email-form-intake.json` (17 nodes) — a standalone background workflow the factory installs into every provisioned system: a daily Schedule Trigger + a Manual Trigger scan Gmail for emails with an attached form (PDF), read each PDF with the **proven multimodal OpenRouter pipeline** (native-PDF `file` content part → `google/gemini-2.5-flash` primary, `anthropic/claude-sonnet-4.5` fallback, `onError: continueErrorOutput`), and in a **PROPOSAL-ONLY** approach (never writes back onto the PDF) extract the form fields, propose per-field values (`source: fixed\|inferred\|missing` + confidence + evidence quote), compose a Hebrew proposal card via `chainLlm`+`lmChatOpenRouter`, run the tg-vision egress validator, and send it to Telegram for the operator. Gmail is read via **HTTP Request → Gmail REST API** with `nodeCredentialType: googleOAuth2Api`, reusing the existing "Google OAuth2 API" credential **without any scope change** — a deliberate divergence from the original plan's Gmail-node approach (the n8n Gmail node requires the single-service `gmailOAuth2` credential type, which `bootstrap-gmail-oauth.yml` does not create; the generic `googleOAuth2Api` is documented for exactly this HTTP-Request use). No custom Docker / pdf-parse / pdf-lib and no third-party PDF/OCR service (the form ID is sensitive — `mistral-ocr` is intentionally avoided). No real PII is committed (the fixed-details profile ships empty; supplied only at runtime). A paired capability skill (`templates/system/.claude/skills/email-form-intake/SKILL.md`, generated from a new `scripts/gen-workflow-skill.sh` table row) keeps the workflow↔skill gate green; `tests/golden/system/MANIFEST.sha256` refreshed; and the new workflow is listed in `monitoring/registry-exempt.txt` (provision-only daily worker — covered collectively by `system-n8n-executions`; intentionally not in the per-workflow `system-n8n-cadence` dead-man map yet, so it won't false-alarm on systems that predate it). Static gates verified locally (jq valid JSON, all 6 Code nodes pass `node --check`, workflow↔skill pair gate PASS). Live proof on `or-edri-4` + the signed E2E proof land in Stage 3. |

## refactor: email-form-intake — read Gmail via a self-minted token (n8n env vars), not n8n's OAuth2 credential (Stage 3, live-found)

| Type | Summary |
|---|---|
| refactor | Stage 3 live-proof finding on `or-edri-4`: **n8n does not use an OAuth2 access token injected via the REST credentials API.** Both `email-form-intake` runs (executions 644 + 645) failed at **List Form Emails** with `NodeApiError: Unable to sign without access token` — even after `bootstrap-gmail-oauth` stored the credential with a freshly-minted `access_token`. n8n only accepts a token from its own OAuth2 "Connect" callback, and the per-system consent `redirect_uri` isn't registered on the shared Google client — so neither the HTTP-Request `googleOAuth2Api` path nor the dedicated Gmail node can drive headless Gmail calls. **Pivot (Or-approved):** mirror the Workspace MCP sidecar, which stays connected headlessly by refreshing the shared token itself. `email-form-intake.json` now mints its OWN Google access token at runtime (a `Mint Gmail Token` HTTP node → `oauth2.googleapis.com/token` with the shared refresh token from n8n **env vars**), and the three Gmail REST calls carry `Authorization: Bearer <minted>` — no n8n OAuth2 credential. `deploy-railway-cloudflare.yml` publishes `GMAIL_OAUTH_CLIENT_ID/CLIENT_SECRET/REFRESH_TOKEN` to the n8n service env (from the system's own SM, masked) + sets `N8N_BLOCK_ENV_ACCESS_IN_NODE=false`. `configure-agent-router.yml` drops the Gmail-credential lookup + `@@CRED_GMAIL_OAUTH_ID@@` (gating now just OpenRouter+Telegram+chat). The earlier `bootstrap-gmail-oauth.yml` token-store tweak was **reverted** (REST-injected tokens are unusable by n8n, so it achieved nothing). `tests/golden/system/MANIFEST.sha256` refreshed. Applied to `or-edri-4` (re-deploy for the env vars + re-configure) and verified live (functional proof + the signed E2E proof). |
