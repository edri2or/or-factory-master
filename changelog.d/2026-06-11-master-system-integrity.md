## fix: db-setup table verification is real and goes RED on a shortfall (master-system-integrity Stage 2, B1)

| Type | Summary |
|---|---|
| fix | The system `configure-agent-router.yml` "installs + runs db-setup, then confirms the bot's 10 Postgres tables exist" — but the confirmation was unreliable and silent. Two real defects: **(1) false-negative** — `_db_ok` filtered executions to the db-setup workflow but had an **unfiltered fallback** (`/rest/executions?limit=1`) that, on a live system with concurrent executions, grabbed a *foreign* workflow's run whose `lastNodeExecuted` was not "Create Tables" → "could not confirm" even when the tables were fine (observed on or-edri-4, configure run 27367116743). **(2) silent** — it only checked that the node *ran*, never *how many tables exist*, and on any miss it emitted a `WARN` while the job stayed **green** ("green ≠ works"). Fix: (a) `db-setup.json` gains a **`Verify Tables`** Postgres node (`Create Tables → Verify Tables`) returning `present_count` of the 10 expected tables; (b) `_db_ok` is rewritten to a `_db_verdict` (`ok` / `fail:<n>` / `inconclusive`) that **deletes the unfiltered fallback**, polls the db-setup-filtered execution (~30s), and requires `lastNodeExecuted == "Verify Tables"` **and** `present_count ≥ 10`; (c) a **confirmed shortfall** (Postgres present, count < 10) sets `DB_VERIFY_FAILED` and fails the job **RED at the end** (`::error:: + exit 1`), non-destructively (router/Telegram/mcp wiring above still ran), while an `inconclusive` read stays a loud `WARN` (never a false-positive RED that would block a clean birth). Proven by a new factory unit test `scripts/tests/db-verify-gate-test.sh` (wired into the **Playground tests** job): a 7/10 fixture → `fail` (RED), a foreign-execution fixture → `inconclusive` (no false RED), 6/6 assertions pass. Golden refreshed. Live positive proof on or-edri-4 (`10/10 tables present`) lands via `refresh-system-agents` + configure; the live negative (dropping a real table) is deferred to a throwaway Day-0 test system. |

## fix: refresh-system-agents produces a live E2E proof so behavior changes can reach existing systems (master-system-integrity Stage 2, B7)

| Type | Summary |
|---|---|
| fix | After the E2E verification gate (PR #416) landed, `refresh-system-agents.yml` could no longer apply a **behavior-bearing** template change to an existing system: it opens a PR in the system repo, and that PR (changing `workflows/n8n/*.json` or `.github/workflows/configure-agent-router.yml`) is blocked by the system's own "E2E verification gate" with no `e2e-proofs/<slug>.json` in the diff. Because the gate's `content_hash` is path-sensitive (`scripts/lib.sh` `e2e_behavior_hash` prepends the file path), a factory-side proof can never satisfy a system gate — the proof must be born in the system repo on the refresh branch. Fix (refresh-system-agents.yml only; reuses the system's shipped `e2e-verify.yml` unchanged): when the synced paths are behavior-bearing **and** the system carries `e2e-verify.yml`, after opening the PR refresh dispatches the system's own `e2e-verify.yml` (`--ref main`, `target_ref=refresh-system-<runid>`, `slug=refresh-system-<runid>`) — which drives a real inbound message, asserts the reply, and commits the signed proof onto the branch — then waits (~15 min budget) for the proof to land, and adds a tiny broker-token follow-up commit to re-trigger runnable CI (the proof commit is pushed with `GITHUB_TOKEN`, whose runs require manual approval and never run unattended). A failed/absent proof fails the refresh (no silent merge of an unproven behavior change). Pre-gate systems (no `e2e-verify.yml`) refresh exactly as before. This restores the standing "iterate on one live system" loop under the new brake — every live application of a behavior change now carries its own real E2E proof. Proven by applying B1 live to or-edri-4 through the new path. |

## fix: db-setup verification reads the executionId from /run (master-system-integrity Stage 2, B1.1)

| Type | Summary |
|---|---|
| fix | B1's verification came back `inconclusive` (not `10/10`) when applied live to or-edri-4 — but `inspect_n8n_execution` proved the db-setup run (execution 93) **succeeded** with `lastNodeExecuted == "Verify Tables"`, i.e. the chain ran and the tables verify. Root cause: `_db_check` found the execution only via `GET /rest/executions?filter={workflowId}` — the same flaky filtered-list lookup that was the original false-negative source — and B1 removed the unfiltered fallback, so when that query missed the just-created run within the poll window the verdict fell through to `inconclusive`. Fix (`templates/system/.github/workflows/configure-agent-router.yml`, db-setup block only): read the **executionId** the `/run` POST returns (`jq '.data.executionId // .executionId // .data.id'` + a grep fallback) and `GET /rest/executions/<id>?includeData=true` **directly** — deterministic, no list-guessing; `_db_check` (filtered list) stays only as a secondary fallback when no id is returned. `_db_verdict` and the `Verify Tables` node are unchanged, so the unit test (`scripts/tests/db-verify-gate-test.sh`) and the RED-on-shortfall path are unaffected. Golden refreshed. Live proof: configure on or-edri-4 now logs `PASS: db-setup verified — 10/10 tables present`. |

## fix: db-setup verification reads every available source (raw-grep) + safe diagnostic (master-system-integrity Stage 2, B1.2)

| Type | Summary |
|---|---|
| fix | B1 and B1.1 both returned `inconclusive` on or-edri-4, yet `inspect_n8n_execution` proved the db-setup run succeeded with `lastNodeExecuted == "Verify Tables"` — the chain runs and the tables verify; only the *retrieval* of the result kept missing (the flaky filtered-list lookup, and a guessed `/run` executionId shape). Key insight: `_db_verdict` is a **raw-text grep** (`lastNodeExecuted…Verify Tables` + `present_count`), not a JSON path — so it works on ANY response that contains the result text. Fix (`configure-agent-router.yml`, db-setup poll only): verdict **every available source** in order, first non-inconclusive wins — (1) the `/run` response itself (often carries the runData inline), (2) the execution fetched by the returned executionId, (3) the filtered-list lookup, last. Robust to whatever shape n8n returns, no guessing. Adds a **safe diagnostic** line (key-paths + count probes only, never values) that fires only if a round is *still* inconclusive, so the exact n8n response shape is revealed for a precise follow-up rather than guessed. `_db_verdict` and the `Verify Tables` node are unchanged (unit test + RED path unaffected). Golden refreshed. Also closed the stale or-edri-4 refresh PR #7 (pre-B7 leftover). Live target: configure on or-edri-4 logs `PASS: db-setup verified — 10/10 tables present`. |
