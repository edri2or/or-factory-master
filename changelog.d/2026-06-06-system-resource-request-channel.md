## feat: system resource-request channel — Stage 1 (deterministic request gate)

| Type | Summary |
|---|---|
| feat | First stage of the system → broker resource-request channel (a human-gated twin of the OIL loop that lets a provisioned system *ask* the broker to create a secret or grant an allowlisted IAM role it cannot do itself). Adds `scripts/validate-system-request.sh` — a pure, network-free deterministic gate that decides whether a request is legitimate before the broker ever acts: validates `request_type` (`secret`/`iam`), refuses control projects + `factory-test-25`, enforces the factory id shape on `system_name`/`gcp_project`, and locks grant targets to the system's own `deploy-sa`+`runtime-sa`. `secret` requests must be a safe id and not a super-credential (mirrors the EXCLUDE set in `copy-generic-secrets.sh`, plus privileged-keyword refusal); `iam` requests are default-deny against a curated, non-escalating role allowlist (owner/editor/`iam.*`/`*.admin`/`serviceusage.*` hard-refused). Proven by `scripts/tests/validate-system-request.bats` (30 cases, auto-discovered by `playground-tests.yml`); shellcheck clean. Behind a human ✅ — no fulfillment yet (Stage 2+). Tracked in `devplans/system-resource-request-channel.md`. |

## feat: system resource-request channel — Stages 2-3 (broker fulfiller + MCP wiring)

| Type | Summary |
|---|---|
| feat | Stage 2 — the broker-side fulfiller. `scripts/fulfill-system-request.sh` performs exactly one privileged action (create a Secret Manager secret shell + grant `secretAccessor` to the system's runtime-sa/deploy-sa, or grant one allowlisted project IAM role to those SAs), idempotent (`describe`/set-semantics) with the `_grant` retry from `grant-secret-accessor.yml`; it never reads or writes a secret value. `.github/workflows/fulfill-system-request.yml` drives it in two phases via `workflow_dispatch`: `register` validates the request (`validate-system-request.sh`) and POSTs the MCP `/system-request-register` to send Or one Telegram approval card (creates nothing); `fulfill` (dispatched only after the ✅) re-validates, performs the action as the broker SA via WIF, then emits an audit event + Telegram confirmation + Linear comment. The target GCP project is resolved **authoritatively** from the system repo's `GCP_PROJECT_ID` variable (never trusted from the request body), so a request can only touch the named system's own project. Not on the `dispatch_workflow` allowlist (reachable only via the gated approval path). |
| feat | Stage 3 — the MCP wiring. New `services/mcp-server/src/system-request.ts` (sibling of `oil-approval.ts`): `dispatchSystemRequest` (dispatch the register phase from a `system.request.*` Linear ticket), `registerSystemRequest` (send the Telegram challenge card; buttons carry only the Linear issue id — `sysreq:`/`sysno:`, state-free), and `handleSystemRequestCallback` (authorise the presser, recover the request from the Linear issue, dispatch the fulfill phase on ✅). `oil-autofix.ts` `handleLinearWebhook` routes `system.request.` tickets here before the OIL triage (no behaviour change to OIL). `index.ts` adds the admin-gated `POST /system-request-register` route and a `sysreq:`/`sysno:` branch in the `/telegram-webhook` router. Proven by `services/mcp-server/test/system-request.test.mjs` (pure decoder/refusal cases); full suite 59/59 green, `tsc` clean. Still code-only — live proof + MCP redeploy are Stage 4. |
