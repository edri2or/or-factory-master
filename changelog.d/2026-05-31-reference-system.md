## feat: reference-system — static golden gate over the templates/system render (Stage 2)

| Type | Summary |
|---|---|
| feat | Stage 2 of the reference-system development — the visible half of the anti-drift mechanism. New `scripts/render-system-golden.sh` re-renders the whole `templates/system/` mould (109 files) with the same fixed 14-var allow-list a real provision uses, replacing each `*.template` with its rendered target name exactly as a provisioned system ends up (deterministic inputs ⇒ no volatile fields to normalise). The chosen `sha256` manifest (vs a full tree copy) keeps storage light while staying byte-exact and catching 100% of drift. |

## feat: reference-system — twin anti-drift CI gate (mould↔golden + allow-list parity) (Stage 3)

| Type | Summary |
|---|---|
| feat | Stage 3 of the reference-system development — the enforcing half of the anti-drift mechanism, a twin of the changelog/devplan gates. New `scripts/check-reference-sync.sh` enforces two invariants over `HEAD~1..HEAD`: **(A) path coupling** — any change under `templates/system/**` (which includes the `deploy-railway-cloudflare.yml` template) must also touch the committed golden `tests/golden/system/**`; always satisfiable because any byte change to the mould moves the `MANIFEST.sha256`, so `scripts/check-system-golden.sh --update` produces the required diff. **(B) allow-list parity** — the envsubst `ALLOWLIST` must stay byte-identical across its three copies (`scripts/render-system-golden.sh`, `.github/workflows/provision-system.yml`, `scripts/tests/validate-templates.sh`); this is the part of a provision change that actually moves the golden, guarded directly so an unrelated `provision-system.yml` edit is never forced to touch the golden. Wired as a new **Check reference golden in sync** step in the existing **Changelog gates** job (`.github/workflows/changelog-check.yml`) — no new required status context. New `scripts/tests/check-reference-sync.bats` (4 tests). Verified locally: no-op pass when nothing relevant changed and the three allow-lists agree; FAIL on a `templates/system/` change without a golden update; PASS when paired with a golden update; FAIL on injected allow-list drift; the three real allow-lists are byte-identical today; full BATS 99/99, `shellcheck --severity=error` + `yamllint` clean. **Plan refinement:** the blunt "any provision/deploy change ⇒ golden touch" coupling from the handoff was replaced by `templates/system/**`-scoped coupling + allow-list parity, because a non-render provision change has no golden to touch and would otherwise be unsatisfiable. |

## feat: reference-system — scheduled drift-reconcile workflow (alert-only) (Stage 4)

| Type | Summary |
|---|---|
| feat | Stage 4 of the reference-system development — the runtime half of the anti-drift mechanism (CI gates only cover code-time; this covers wall-clock drift on the live system). New `.github/workflows/reference-system-reconcile.yml` (cron every 6h at `:30`, staggered off factory-health `:00` / system-runtime `:15`, plus manual `workflow_dispatch`), modelled on `system-runtime-audit.yml`. It reads `reference-system/config.yml` via `scripts/reference-config.sh` and, **only once provisioned** (`provisioned: true`, set in Stage 0), checks two signals and emits one event through `scripts/emit-event.sh`: **mould drift** — `git diff built_from_commit..main -- templates/system .github/workflows/provision-system.yml` (the standing system is stale if the mould moved since it was built) — and **liveness** — a `/healthz` 2xx probe. In sync + healthy ⇒ `factory.reference_reconcile.ok` (info → Axiom); otherwise ⇒ `factory.reference_reconcile.drift` (error + action_required → Axiom + Telegram + Linear). Until provisioned it is a **clean no-op**: it reads the config, logs, and exits without touching GCP/WIF. **Alert-only by design — it never auto-rebuilds**; re-provision/redeploy is a costed move that stays human-gated per CLAUDE.md's "one rule", so the optional auto-dispatch-rebuild from the handoff was deliberately dropped. Verified locally: `yamllint` + `actionlint -shellcheck "shellcheck -S error"` clean; the not-provisioned no-op resolves `skip=true` against today's config; the decision logic emits `ok` only for `in_sync`+2xx and `drift` for the other three cases; and the `git diff` mould signal correctly distinguishes an unchanged mould (`in_sync`) from a changed one (`stale_mould`). The live ok/drift path is exercised after Stage 0. Registered in `monitoring/watchdog-registry.json` (cron-workflow, `gh-run-freshness`, 9h tolerance) so the meta-monitoring watchdog proves the cron actually keeps running — the new-surface CI gate (`check-watchdog-registry-updated.sh`) required it. |
