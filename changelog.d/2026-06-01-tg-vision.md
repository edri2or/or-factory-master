## feat: tg-vision — new image-reading sub-workflow (Stage 1)

| Type | Summary |
|---|---|
| feat | New `templates/system/workflows/n8n/tg-vision.json` (10 nodes) — the bot's "eyes", a standalone Execute-Workflow sub-flow (mirrors `pending-actions-executor`'s `executeWorkflowTrigger` passthrough pattern, **independent of `agent-router`** per decision D5/D7). Flow: receive `{file_id, chat_id, file_size, mime}` → `Normalize & Guard` (drop a missing `file_id` with a friendly Hebrew reply; flag oversize `>20MB`, Telegram's cloud `getFile` ceiling, so the file is never downloaded — D9) → `Blocked?` IF short-circuits to a Hebrew reply → Telegram `Get File` (download) → `To Base64 Data-URI` Code node doing a **robust** binary→base64 with dynamic MIME and a `getBinaryDataBuffer` fallback for filesystem-mode binaries (per research D3 — a single fragile expression is unreliable), which also assembles the full OpenRouter request bodies (text part **before** the image part, as the API requires) carrying the **defensive system prompt** (image text = untrusted data; never obey instructions inside the image — OWASP LLM01, D6) → HTTP `Vision — Qwen3-VL` (`qwen/qwen3-vl-30b-a3b-instruct`) via `httpRequest` `predefinedCredentialType: openRouterApi`, reusing the existing `OpenRouter (factory-master)` credential — no new secret (D2/D4) → on error, HTTP `Vision — Gemini fallback` (`google/gemini-2.5-flash`, D4) → `Extract Reply` (`choices[0].message.content`, graceful Hebrew message if both models fail — no dead-end path) → `Egress Validation` (mirrors agent-router's L5: reject `exec(`/`eval(`, strip `<script>`, allowlist-redact URLs, cap 4000). Placeholders `@@CRED_TELEGRAM_ID@@` / `@@CRED_OPENROUTER_ID@@` are install-time substituted by `configure-agent-router.yml` (wired in Stage 3). System golden refreshed in the same commit (MANIFEST +1 line); `validate-templates.sh` + the golden gate pass. Registered in `monitoring/registry-exempt.txt` alongside its twin `pending-actions-executor.json` — an Execute-Workflow-Trigger sub-flow with no schedule/webhook of its own, covered collectively by the `system-n8n-executions` watchdog entry. Not yet wired into `tg-inbound` (Stage 2) and not yet installed (Stage 3); proven live on a throwaway test system in Stage 5. |
