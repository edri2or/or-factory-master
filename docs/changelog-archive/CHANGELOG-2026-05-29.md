## Stage 115 ‚Äî feat: Phase F Gap 5 ‚Äî real OpenRouter cost in spend_log via key-usage polling

| PR | Type | Summary |
|---|---|---|
| TBD | feature | Closes Gap 5 of `docs/phase-f-handoff.md` ¬ß2. `spend_log.cost_usd` was hardcoded `0` (only writer = `tg-proactive`'s "Log Spend" node), so the daily 24h spend summary always showed $0. The handoff's suggested fix ‚Äî a Code node reading `$('OpenRouter Chat Model').last().json.usage` ‚Äî is **infeasible**: n8n's `lmChatOpenRouter` sub-node `response.tokenUsage` is silently dropped and unreachable downstream (`"No path back to the referenced node"`; confirmed by n8n#26302 / n8n#21673 + docs), for both `chainLlm` and `agent` nodes. First run seeds the baseline WITHOUT logging history; negative deltas clamp to 0 (month/key-reset guard); unreadable usage logs nothing and self-heals next run. `db-setup.json`'s "Create Tables" node gains `CREATE TABLE IF NOT EXISTS spend_track_state` (singleton `BOOLEAN` PK, `NUMERIC(12,6)`) + the table appended to the verification SELECT list. `tg-proactive.json` loses the hardcoded `cost_usd 0` "Log Spend" node + its connection (`Send Proactive` becomes terminal); its unchanged 24h `SUM(cost_usd)` reader now reflects REAL spend. `configure-agent-router.yml` gains a soft-fail install block (section 5e) that upserts+activates spend-track, gated on Postgres + OpenRouter (NOT Telegram/chat_id), plus a job-summary row. Idempotent + soft-fail (every PG/HTTP node `onError:continueRegularOutput`; a separate workflow sharing no nodes with the chat path ‚Üí can never affect the bot). Per CLAUDE.md `templates/system/` edits reach only newly-provisioned systems; verification = re-run `configure-agent-router.yml` on `factory-test-tgbot16` (idempotent), trigger spend-track (seed ‚Üí delta>0 after a real chat message), confirm a non-zero `openrouter-key` `cost_usd` row + tg-proactive `spend_24h>0` via `inspect_n8n_execution`. ## Stage 113 ‚Äî fix: postgres_named_query ‚Äî drop specifyInputSchema, trigger back to passthrough (proven on tgbot15 with the operator sending a real message)

| PR | Type | Summary |
|---|---|---|
| TBD | fix | Stages 110/111/112 all chased the wrong path: with `executeWorkflowTrigger` in `workflowInputs` mode + the `toolWorkflow` node carrying any flavour of `specifyInputSchema` (`manual` JSON Schema with enum, or `fromJson` example), n8n 1.121's bridge between LLM args and the trigger's `$json` shipped `query_name: null` regardless of what the LLM provided. Verified deterministically by two diagnostic rounds on `factory-test-tgbot14` (DEBUG1, after Stage 111's `workflowInputs`) and on `factory-test-tgbot15` (DEBUG2, after Stage 112's `schemaType:"fromJson"`): both surfaced identical evidence ‚Äî `Keys at root: ["query_name"]`, `typeof query_name: object` (JS quirk: `typeof null === "object"`), `_raw_input: {"query_name": null}`. The schema ‚Üí Zod ‚Üí tool-args ‚Üí trigger-pickBy pipeline in toolWorkflow v2.1 simply does not propagate the LLM's structured value to the sub-workflow. **DEBUG3 ran with `specifyInputSchema:false` (drop schema entirely) + trigger back to `passthrough`** ‚Äî tgbot15 then produced a real bot reply ("your style profile is still empty"), and `inspect_n8n_execution` confirmed execution 28 terminates at **`Format Output`** (the success path), not `Unknown Query`. That's the first end-to-end success for the tool. Fix in main: (1) `templates/system/workflows/n8n/postgres-named-queries.json` ‚Äî trigger back to `inputSource:"passthrough"` (the `workflowInputs` config was load-bearing only for Stages 110-112's schema dance, now reverted), Normalize Code node updated to extract `query_name` from `$json.query` as a raw string against the 4-value whitelist (also tries JSON.parse for safety). (2) `unknown-agent.json` + `ops-agent.json` ‚Äî `postgres_named_query` toolWorkflow nodes lose `specifyInputSchema`, `schemaType`, `jsonSchemaExample`. Tool description rewritten to be unambiguous about the call shape ("Call this tool with ONE input ‚Äî a single string that is EXACTLY one of these 4 query names, do not wrap it in JSON or any object, just send the raw string"). Inline code-comment on the Normalize node records the 3-stage diagnostic so a future reader sees why this specific shape was chosen. Per CLAUDE.md `templates/system/` edits reach only newly-provisioned systems; tgbot15's diagnostic copy is already shaped this way (the success was on tgbot15 *with* the diagnostic Telegram node ‚Äî that node lives in tgbot15's own repo copy and won't migrate to other systems).
