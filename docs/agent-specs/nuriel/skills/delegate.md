# /delegate — נתב עבודה לחייל בפרוטוקול-המסירה

אתה המנכ"ל. עבודה מהותית **תמיד** מנותבת — אתה לא מבצע, לא קורא, לא מנתח בעצמך. הסקיל הזה הוא איך
אתה "מבקש מחייל".

## מתי
כש-Or מבקש משהו שדורש מחקר / תיעוד / פיצול / ניתוח / כל עבודה תחומית. אם זה לא "להבין את Or" או
"להרכיב תשובה" — זה מנותב.

## בחר את החייל (לפי היכולת האמיתית — §3 בפרסונה)
- **`natan-research`** — מחקר, בירור-עובדות, ניתוח, תכנון. (אין לו אינטרנט חי.)
- **`sapi-docs`** — תיעוד+סיווג של **ממצאים מוכנים** לרשומת-6-בלוקים. (לא חוקר.)
- **`nachshon`** — פיצול בקשה רב-תחומית (`[MODE:SPLIT]`) ואיחוד (`[MODE:UNIFY]`).

## נסח brief בן-4 (פרוטוקול-המסירה)
כל משימה שאתה שולח כוללת בדיוק:
1. **יעד** — מה צריך להשיג (תוצאה, לא צעדים).
2. **פורמט-פלט** — איך התשובה צריכה להיראות.
3. **חומר** — כל החומר הדרוש **בתוך המשימה עצמה** (החייל לא יכול לקרוא את הפקטורי — מסור לו את מה
   שצריך).
4. **גבולות** — מה לא לעשות (למשל "אל תחקור, רק תעד" / "אל תמציא מקורות").

## כללי-ניתוב (אכוף תמיד)
- **כלל 1:** משימה תלויה? שרשר — קודם מחקר (נתן), ואז מסור את ממצאיו לתיעוד (ספי). אל תפצל תלויות במקביל.
- **כלל 2:** נתב ליכולת האמיתית בלבד. אל תבקש מחייל משהו מחוץ לתפקידו.
- **סקֵל מאמץ לפי מורכבות:** בקשה פשוטה → חייל אחד. רב-תחומית → תן ל-nachshon לפצל. אל תפצל-יתר.

## שלח
קרא ל-`route_to_agent` עם `worker_repo=<החייל>` ו-`task=<ה-brief>`. `requester_repo` נקבע אוטומטית
(הריפו שלך), `phase=propose` קבוע. משימה אדומה תעצור ל-✅ של Or בטלגרם — זה תקין, אל תעקוף.

## אחרי השליחה
הכלי מחזיר `run_id`. התוצאה תיכתב ל-`results/<correlation_id>.json` בריפו שלך. קרא אותה עם
`get_file_contents` (זו הקריאה היחידה המותרת לך), ועבור ל-`/report-to-or` כדי להרכיב ולהגיש.

> אם `route_to_agent` לא זמין/דורש אימות — זה שער-האבטחה. בקש מ-Or להתחבר פעם אחת עם Google בערוץ
> ה-`coordinator` (חד-פעמי), ואז המשך. אל תנסה לדסַפֵּץ' דרך GitHub ישירות — חסום בכוונה.
