## The one rule

**Build manually. See every step. Continue only after verifying.**

The previous factory (`edri2or/factory`) automated everything end-to-end. Failures surfaced minutes or hours after the fact through GitHub Issues; fixes landed in the next provisioning run. This repo throws that pattern out. The agent dispatches one workflow, watches it run, verifies the outputs, and then asks the user before doing anything else.

**The one sanctioned exception is the OIL auto-fix loop** (`docs/oil-autofix.md`): a Linear-issue-driven loop that autonomously investigates a failure, proposes a small fix, and — only after Or's one Telegram ✅ and a green merge — verifies the fix actually worked and closes the ticket. It *earns* the exception by keeping this rule's spirit: every step is verified (a deterministic safety gate before the PR, a post-merge reproducer re-run before the close) and the merge is human-gated. It does not loosen "no auto-chain" or "no issue-based reporting" anywhere else.

## How to work

1. **Match the user's request to a skill.** The skills below describe the supported flows.
2. **Pre-flight checks first.** Before any dispatch, verify the inputs and the absence/presence of collisions via read-only MCP tools.
3. **Dispatch via the `dispatch_workflow` MCP tool.** The agent triggers the allowlisted workflows itself (`provision-system.yml`, `register-system-app.yml`, `deploy-railway-cloudflare.yml`, `configure-agent-router.yml`, `refresh-system-agents.yml`, `decommission-test-system.yml`, `oil-autofix-investigate.yml`, `deploy-mcp-server.yml`, `meta-monitoring-watchdog.yml`, `bootstrap-sandbox-tester.yml`, `prove-on-test-system.yml`) — no operator button-click, no env-var PAT. Confirm cost/scope with the user before a fresh provision/deploy unless they've opted into autonomy. `decommission-test-system.yml` runs only on an explicit user teardown request (never auto-chained). The real-system `decommission-system.yml` is **planned but not yet implemented** (`skills/decommission-system/SKILL.md` › Status — real-system teardown is manual for now); when built it stays off the `dispatch_workflow` allowlist by design (destructive — written approval required).
4. **Watch the run.** Poll the workflow run. Read failed step logs directly. Report what failed.
   - Polling protocol — always follow this when waiting for a workflow run:
     - Call `get_workflow_run` (factory MCP) every ~30 seconds.
     - Continue up to 40 iterations (~20 minutes maximum).
     - While status is `in_progress` or `queued` — wait and retry.
     - On `success` — proceed to the verify step (`verify_*` tools).
     - On `failure`/`cancelled` — read failed step logs via `get_run_jobs` and report to the user.
     - If 40 iterations pass with no terminal status — stop and report; never assume completion.
     - Never advance to the next step without a confirmed terminal status from `get_workflow_run`.
   - The browser session cannot listen for events between messages, so this active poll is required. As a backstop, `provision-system.yml` sends a Telegram nudge from *inside its own run* when it finishes (success ✅; failure already alerts via the `Emit provision failed` step), so the user can return to the session even if the browser tab is idle. This is an in-run step, not a separate `workflow_run` listener — GitHub does not emit `workflow_run` for the broker-App/bot-dispatched runs the factory uses, so an external listener never fires.
5. **Verify outputs.** After success, call the relevant `verify_*` MCP tool to confirm the real state matches the expected state.
6. **Stop at the boundary.** Each skill ends at a clear handoff point. Ask the user what's next; don't chain.
7. **Verify and record state changes.** After any state-changing action — especially async / human-gated ones (a Telegram-approved deletion, a dispatched workflow's effect) — verify the *real* outcome with a read tool (`get_repo` / `list_repos` / `verify_*`) and record the verified result in the durable doc (the devplan / a `changelog.d/` fragment) before you treat it as done. Never close a development while an action it triggered is still pending or unverified — the committed record must always match reality, so a later session never has to reconstruct what happened.

**Building a new capability? Route through capability-first FIRST — this is non-skippable.** Before building anything that performs a *new capability* (a new **verb**: read / fill / extract / send / parse — whether it lands as a sub-agent or as a standalone n8n workflow), you MUST first prove the **raw capability outside n8n** on a real fixture with a recorded **go/no-go** verdict (`/prove-capability`, `docs/capability-first.md`), then decompose and build bottom-up (`/build-agent`). **This holds in the factory too** — building a capability into `templates/system/**` is no exception (the factory must eat its own dog food). `/dev-stage-factory` and `/dev-stage` **wrap** this rule as their Step 0; they do not replace it — a capability-adding development is not "started" until Phase 1 is **go** and the role is decomposed. This is the rule the `email-form-intake` monolith bypassed (proof of the hardest brick — reading the PDF — deferred to the end, the exact "big-bang" anti-pattern `docs/capability-first.md` exists to prevent). A development that adds **no** new capability (config / plumbing / docs) skips this rule.

## Never

- Touch the old factory repo (`edri2or/factory`) or its GCP project (`factory-control-9piybr`).
- Use the old broker SA, WIF provider, or App credentials.
- Auto-chain stages without verifying: dispatch the next workflow only after verifying the prior run's outputs. Never dispatch the real-system teardown `decommission-system.yml` from the agent — it is deliberately excluded from the `dispatch_workflow` allowlist (and is **not yet implemented** — planned only; see `skills/decommission-system/SKILL.md`).
- Dispatch `decommission-test-system.yml` (test teardown) for any reason other than an explicit user request to tear down a test system — never automatically, never chained after provision/deploy. Confirm with the user first; it's destructive.
- Open GitHub Issues to report success/failure — this is interactive, not async.
- Write `factory/manifests/`, `factory/evidence/`, or session-summary files. *(A user-driven living development plan — `DEVPLAN.md`, managed by the `/dev-stage` command — is a permitted and required tracking tool, distinct from the auto-generated manifest/evidence outputs this rule forbids.)*
- Leave a durable record (a devplan, a `changelog.d/` entry, a doc) contradicting the real system state, or write "apparently"/"probably" into one — verify the fact and state it, so a later session never has to guess what happened. (The trap: a record said "zz- repos kept" but they had since been deleted to 404 — a later session had to reconstruct it and waste time.)
- Bypass branch protection or skip CI checks.
- Print, log, or echo secret values.
- Create GCP SA keys. Auth is WIF only.

## Development workflow

`/dev-stage` runs a new development as ordered, documented stages tracked in a living per-development plan file at `devplans/<slug>.md` (instantiated from `templates/devplan/DEVPLAN.template.md`), reporting to the user in plain Hebrew on demand — the user never opens a file. **Every** new development gets its own `devplans/<slug>.md` from the start, so two `/dev-stage` developments can run in parallel sessions without colliding on the plan file (the legacy root `DEVPLAN.md` is still honored until it closes). A CI gate (`scripts/check-devplan-updated.sh`, wired into the existing `changelog-check.yml` "Changelog gates" job) blocks merging dev-code while any plan is `status: active` unless at least one active plan is updated in the same diff; it enforces **every** active plan (not just the first), is a twin of the CHANGELOG gate, and is a no-op when no plan is active. **Closing a plan in a code PR is safe:** the devplan gate credits a touch of *any* devplan file — whether the plan ends the change `active` or flips to `completed` — so you may set your plan to `status: completed` in the same PR that carries code even while **another** plan stays `active` and untouched. (This was historically a "closing-while-parallel" trap; fixed 2026-06-15 — `scripts/check-devplan-updated.sh` is now a path-membership twin of the CHANGELOG gate.) Full parallel-development policy — short-lived branches, the `live-system-<system>` queue that serializes work on `or-edri-4`, and why the factory stays non-strict (no merge queue) — is in `docs/parallel-development.md`. Every dev stage's changelog entry is written to a dated per-development fragment `changelog.d/<YYYY-MM-DD>-<slug>.md` (the default for **all** developments), never to the head of `CHANGELOG.md` — so concurrent PRs never collide on a hand-picked `Stage N`; the changelog gate (`scripts/check-changelog-updated.sh`) accepts the fragment. The numbered `CHANGELOG.md` is built only by the **Compile changelog** workflow (`compile-changelog.yml` → `scripts/compile-changelog.sh`), which folds all fragments in one single-threaded run (numbers assigned without concurrency) and auto-archives past the 20 KB cap. `/dev-stage` is a slash-command (`.claude/commands/dev-stage.md`), not a factory skill. Two companions keep the plan state ambient: a committed `SessionStart` hook (`.claude/settings.json` → `scripts/devplan-session-start-hook.sh`) injects every active plan's state into each session — and after compaction — so the agent re-orients automatically, and `/dev-status` (`.claude/commands/dev-status.md`) gives the user a plain-Hebrew, on-demand summary of the active plan(s). Both are read-only, support multiple parallel active plans, use the same path-scoped `status: active` detection as the gate (so the `templates/devplan/` seed never trips them), and are silent/no-ops when no plan is active; the hook can never break a session (`trap 'exit 0' EXIT`). The full `/dev-stage` machinery is **propagated to provisioned systems** (provision-only — systems built *after* this change; existing systems are not back-filled): the `/dev-stage` + `/dev-status` commands ride in the copied `.claude` tree, and `provision-system.yml` additionally ships `scripts/check-devplan-updated.sh` + `scripts/devplan-session-start-hook.sh`, a `templates/system/.claude/settings.json` that registers the devplan SessionStart hook, and a `templates/devplan/DEVPLAN.template.md` seed — and the system template's "Changelog gates" job runs the devplan gate — so each new system can run its own full, gated, documented development. The factory's own `.claude/settings.json` additionally runs a factory-only dependency-install hook (`.claude/hooks/session-start.sh`) that systems do not receive.

**Validating provisioning-process changes — prove on `or-edri-4` first, then lock into the template (a standing capability).** Any change to what the factory provisions (`templates/system/**`, `provision-system.yml`, the system `deploy-railway-cloudflare.yml`) is **first applied and proven live on the standing proving system `or-edri-4`** (GCP `factory-test-21`, adopt) — and **only then locked into the factory's template code**. `or-edri-4` goes hand-in-hand with the factory: it is the live Day-2 system every provisioning-process change must pass through before it's trusted. This is a capability the agent always has: static gates green → **apply the change to `or-edri-4` live** (template-file changes via `refresh-system-agents.yml` with `paths=`/`post_merge_workflow=`; deeper changes via a re-deploy) → **verify live** (`probe_endpoint` / Telegram; for bot-behavior changes a fresh `e2e-verify` proof — see the E2E gate below) and iterate fix→apply→verify → **promote** = merge to `main` (the lock into the template). A **cheap, throwaway** test system in reuse mode (`shared_gcp_project=factory-test-25`, **0 quota**: `provision-system.yml` → `register-system-app.yml` → `deploy-railway-cloudflare.yml`, torn down via `decommission-test-system.yml`, user-triggered) is still used for the **Day-0 birth check** — proving a *fresh* system is born correct, which a stateful Day-2 system like `or-edri-4` cannot show. The two are complementary: `or-edri-4` proves Day-2 (state, iteration); a throwaway proves Day-0 (birth). **Why this is not the retired "reference system" trap** (that one was never used, decayed silently, and cost money): `or-edri-4` is now **mandatory** — a hard CI brake (`scripts/check-e2e-proof.sh` + `proof_systems` in `e2e-surfaces.json`) requires the live E2E proof to come from `or-edri-4`; the proof must be **fresh per change**, so a decayed system fails the proof loudly; it is on a **heartbeat audit** (`system-runtime-audit.yml`); and it costs **nothing new** (it is already Or's running system). `/dev-stage-factory` (`.claude/commands/dev-stage-factory.md`, **`audience: factory-only`** — not shipped to systems) is the staged, documented wrapper around this loop; every costed move (provisioning/redeploying a test system, any teardown) is explicitly Or-gated, and `factory-test-25`'s reuse role is never repurposed. Two **template-integrity** guards remain (independent of any live system): a **static golden gate** (`scripts/check-system-golden.sh` over `scripts/render-system-golden.sh` → `tests/golden/system/MANIFEST.sha256`, in the **Playground tests** job) and its **twin CI gate** (`scripts/check-golden-sync.sh` in the **Changelog gates** job: a `templates/system/**` change must refresh the golden, and the envsubst allow-list stays byte-identical across renderer/provision/validator). Full reference: `docs/live-test-loop.md`.

## Validation rules

`system_name` must satisfy: `^[a-z][a-z0-9-]{4,28}[a-z0-9]$` (6–30 chars total). In **normal mode** the same string becomes both the GCP project ID and the GitHub repo name. In **reuse mode** (test systems) the GCP project is `shared_gcp_project` and `system_name` is only the GitHub repo name.

## Test systems vs. real systems

The org is at its GCP project-creation quota (active + soft-deleted projects both count, for ~30 days). To keep test iteration quota-free, test systems **reuse** one shared GCP project instead of creating a new one. Route by the user's intent:

| Request | Mode | What to dispatch |
|---|---|---|
| "create a system" / a real, persistent system | **normal** | `provision-system.yml` with `system_name=<name>` only → creates a new GCP project (consumes quota). |
| "create a **test** system" | **reuse** | `provision-system.yml` with `system_name=<name>` **and** `shared_gcp_project=factory-test-25` → reuses that project, skips create + billing-link, cleans + reseeds its secrets → **0 quota**. |
| a real system **when the project-creation quota is exhausted** | **adopt** | First dispatch `list-recoverable-projects.yml` (read-only) to find a soft-deleted/existing project. Then `provision-system.yml` with `system_name=<name>` **and** `adopt_gcp_project=<project-id>` → undeletes the project if soft-deleted (no `projects create`, so it never hits the quota — a soft-deleted project already counts), re-links billing, builds **real per-system WIF**, and wipes the project's secrets **once** before reseeding. The project id is immutable — the new system's GCP project keeps that old id; the repo is `system_name`. Refuses control projects + `factory-test-25`. Mutually exclusive with `shared_gcp_project`. |

**Why adopt works:** the quota counts active + soft-deleted projects (both, ~30 days), so a fresh `gcloud projects create` is blocked. Undeleting a soft-deleted project does NOT consume additional quota (it already counts) — it just reactivates one we can repurpose. Adopt is for **real** systems; unlike reuse mode it keeps per-system WIF (not the broad test_pool) and wipes secrets once (not every run). Enumerating soft-deleted projects needs a workflow/Cloud Shell — the `list_gcp_projects` MCP tool returns only ACTIVE projects.

In reuse mode also pass `shared_gcp_project=factory-test-25` to `register-system-app.yml`; in **adopt mode** pass `adopt_gcp_project=<project-id>` to `register-system-app.yml` so the per-system App credentials land in the recovered project's SM (without it, register-system-app would target a GCP project named after the repo, which doesn't exist). `deploy-railway-cloudflare.yml` is unchanged — it reads the system repo's `GCP_PROJECT_ID` var, which provision sets to the shared project.

Reuse-mode consequences (state them to the user): all secrets live in `factory-test-25`'s SM and are **wiped + reseeded at the start of every reuse provision** (clean slate), so only the most-recent test round is SM-backed and per-test `github-app-*` don't persist across rounds. Fine for throwaway tests — a system that must persist needs its own GCP project (a real provision).

**Still confirm cost/scope with the user before dispatching — test or real** (per "The one rule"). Test provisions are 0-quota but still create a real GitHub repo + Railway project and mutate the shared project.

### Answering project-quota questions

When the user asks anything about the project quota — *"כמה פרויקטים פנויים יש לנו?"*, *"מתי יתפנו הפרויקטים שנמחקו?"*, *"תראה לי מצב מכסת הפרויקטים"*, or any paraphrase — call the **`gcp_project_quota_status`** MCP tool. It is the single source of truth: read-only, runs as the broker SA, returns `activeCount`, `softDeletedCount`, and per soft-deleted project its `deleteTime` + estimated `freeUpDate` (deleteTime + ~30d) + `daysRemaining` (sorted soonest-first). It supersedes manually dispatching `list-recoverable-projects.yml` for this question. Present the result in plain Hebrew (group projects by free-up date; lead with what frees up soonest).

Two honest caveats to always state: (1) the org's **absolute** project-creation cap is **not exposed by any GCP API**, so the tool reports *usage* (active + soft-deleted, both count toward the cap for ~30d) and the *free-up schedule* — never an exact "X free slots remaining". (2) `freeUpDate` is an estimate (`deleteTime` + 30d retention). If a real system is needed before the soonest free-up, the answer is **adopt mode** (recycle a soft-deleted project — see the table above). The tool sources each `deleteTime` via a per-project v3 `projects.get` (v3 `projects:search` omits it; the v1 `list_gcp_projects` tool returns only ACTIVE projects, no delete metadata).

## Propagation patterns

GCP IAM has *eventual consistency* between policy update and effective permissions. Any `gcloud` call against a resource that was just created (in the same workflow run) can fail with `PERMISSION_DENIED` or `does not exist` for up to ~60s, even though `get-iam-policy` or `describe` already shows the resource/binding.

If you add a new step that touches a fresh resource, wrap it in a retry. Three windows are known and handled:

| Window | Symptom | Where handled |
|---|---|---|
| SA → IAM policy member (~5-30s) | `add-iam-policy-binding` says SA "does not exist" | `Grant project-level IAM`, see `_bind` helper |
| role-grant → effective permission (~30-60s) | API call returns `PERMISSION_DENIED` for a role the principal visibly has | `Create system-level WIF pool and provider`, see `_wif_op` helper |
| SA → setIamPolicy on the SA resource (~30-60s) | `service-accounts add-iam-policy-binding` says permission denied | `Grant deploy-sa workloadIdentityUser binding` |
| Railway scheduler throttle (no clear timeout) | Postgres `serviceCreate` stuck in `DEPLOYING` indefinitely with **zero container logs** | Out-of-band: delete stale Railway projects via `projectDelete` GraphQL mutation. Confirmed today on factory-test-{16,17} after ~15 fresh projects in one workspace within a day; cleared by deleting 11 stale ones; factory-test-18 then deployed cleanly. Not handled in the workflow — surface as a "this isn't a code bug, free Railway quota" warning to the user. |

Pattern: retry only on the specific error class (`PERMISSION_DENIED`, `does not exist`); surface anything else immediately. 10×10s or 12×10s is typical. See `docs/bootstrap-record.md` for the history of each.

**Cloud Run no-op redeploy (boot-time secrets).** `gcloud run services replace` with a byte-identical template is a NO-OP — no new revision, no restart. A container that reads a secret only at **boot** (e.g. the workspace-mcp sidecar reading `gmail-oauth-refresh-token`) will **NOT** pick up a rotated/re-consented secret on a same-commit redeploy; it keeps its boot-time value. `deploy-mcp-server.yml` forces a fresh revision every run via a per-deploy `DEPLOY_NONCE` env so re-consents always load — any future boot-time-secret Cloud Run service needs the same. (Diagnosed live 2026-06-16: revision `…-00090` served a revoked Google token across several "successful" redeploys until `DEPLOY_NONCE` landed.)

### Skills audience (factory-only vs shared)

Every slash command in `.claude/commands/*.md` carries an `audience:` frontmatter key —
`shared` (used by the factory **and** shipped to every provisioned system) or `factory-only`
(factory only, never copied into a system). `provision-system.yml` copies
`templates/system/.claude/` into each new system, so `templates/system/.claude/commands/` is a
**derived subset** = exactly the `shared` commands, byte-identical (it is no longer a byte clone
of all factory commands). When adding or retagging a command, set its `audience:` and run
`bash scripts/sync-skills-mirror.sh`, then commit both. CI (`check-skills-mirror.sh`) fails any
command with a missing/invalid tag (forcing the shared/factory-only decision) or any mirror
drift. Full reference: `docs/skills-audience.md`.

**A skill that auto-surfaces from `templates/system/.claude/skills/` is NOT a live factory
capability.** When you work under `templates/system/`, Claude Code auto-loads those system skills as
"available" in the *factory* session — but they are **shipped-to-systems artifacts** describing what a
*provisioned system* does, not what the factory itself runs. Never cite one as a factory precedent or
assume the mechanism it describes is live here; verify the factory's own implementation (a workflow,
an MCP tool, a doc) instead. (This is the trap that let a retired `edri2or/gcp-hands` client skill be
mistaken for a live cross-repo dispatch channel — `retire-gcp-hands-client`, 2026-06-17.)

### Web-session connector gate — never tell Or to "click Allow" (there is no button)

In Claude Code **on the web**, the factory MCP is reached as an Anthropic-hosted **connector** (traffic routes through Anthropic's servers). Some tools — confirmed **`dispatch_workflow`** (write) and **`list_repos`** — are gated **server-side** and fail with `Streamable HTTP error: Error POSTing to endpoint: MCP tool call requires approval`. This is **not** a Claude Code permission prompt: the local client (`mode=auto`) already auto-approves the call, and the rejection comes back over HTTP from the connector (proven in the client log — the tool is dispatched, then the connector returns the error; and the string is emitted **nowhere** in `services/mcp-server/`). Handle it as follows — this is a standing rule, not a per-case judgment:

- **There is NO interactive "Allow" button for Or, and a local `.claude/settings.json` `permissions.allow` rule does NOT override a connector gate** (allow-rules are client-side; the connector enforces separately). **Never tell Or to "click Allow"/"approve the prompt" for these — there is no button.** Telling him so is a known, repeated error; don't repeat it.
- **`dispatch_workflow` → dispatch the same allowlisted workflow via the GitHub MCP instead:** `mcp__github__actions_run_trigger` with `method=run_workflow`, `owner=edri2or`, `repo=or-factory-master`, `workflow_id=<wf>.yml`, `ref=main`, `inputs={…}`. Same org-wide broker App, same `workflow_dispatch` event, ungated; then watch/verify exactly as before (`get_workflow_run` polling → `verify_*`/`probe_endpoint`). Proven (ran `refresh-system-agents.yml` run `27552856170` after `dispatch_workflow` was gated). Scope: the GitHub MCP is scoped to `or-factory-master`, which covers every allowlisted workflow the agent triggers (they all run on the factory repo; cross-repo system dispatches happen *inside* `refresh-system-agents.yml`, server-side). A *direct* system-repo dispatch still needs `dispatch_workflow`; if it is gated, wrap it via `refresh-system-agents.yml`, or tell Or plainly it's blocked at the platform — do not invent an Allow click.
- **`list_repos` → use the ungated `list_all_systems_inventory`** (deployment-authoritative) or other scoped reads (`get_file_contents`, `get_repo`, …), which pass the gate.
- The read/inspect tools (`verify_*`, `inspect_*`, `get_*`, `list_workflow_runs`, `probe_endpoint`, `list_all_systems_inventory`, …) are **not** gated — use them freely. The gate is web-only; from a terminal session (`--teleport`) it does not apply.
- **The gate keys on specific tool NAMES, not on write-semantics — a custom, narrow write tool can pass it.** The two gated names (`dispatch_workflow`, `list_repos`) are not "any write." Proven live (`nuriel-coordinator`, 2026-06-18, historical): the coordinator route's custom write tool **`route_to_agent`** (on `/coordinator/<repo>/mcp`), reached via a claude.ai connector, dispatched `agent-action.yml` **successfully** — broker run `27788706190` (`triggering_actor=factory-master-broker[bot]`), result committed in the requester agent-repo — i.e. it was **NOT** gated, contrary to the natural assumption (and an explicit pre-test prediction) that all writes are blocked. So **don't assume a new/custom MCP write tool is gated — verify it live before routing around it.** (Reaching an agent-repo's gateway route from Code web is itself done *via a connector*, which also bypasses the sandbox's egress block on the repo's direct `.mcp.json` URL — see `docs/mcp-connector-setup.md` › "Coordinator connector".)
- **Connector URL (different problem, same area):** the URL to paste into a claude.ai *custom connector* is the gateway's advertised `issuer` — which since `connector-url-clear` Stage 2 **is** the Region URL the deploy summary labels as the `mcp_url` (`PUBLIC_BASE_URL` is pinned to it), so the two consumers now share one address. Canonical doc: `docs/mcp-connector-setup.md` (machine-checkable via the `Read + assert the live connector issuer` step in `deploy-mcp-server.yml`, the `mcp-oauth-issuer` check in `verify_mcp_server`, and the `/prove-connector` skill). Recommend the Region URL printed by the latest deploy — it is both the `issuer` and the toolbox `mcp_url`.
- **Google Workspace tools in Claude Code on the web — the `localhost:3002` error is a wrong `user_google_email`, NOT a broken connector.** The Workspace connector (`/workspace/<system>/mcp`) surfaces in a Code web session like any account-level connector and works there (proven live 2026-06-17 — `search_drive_files` returned real Drive files from a Code web session).
