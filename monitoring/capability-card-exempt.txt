# Capability-card exemptions — operable n8n workflows in the mould that need NO
# Phase-1 Capability Card, because they introduce no NEW external capability that
# capability-first (docs/capability-first.md) exists to de-risk.
#
# Read by scripts/check-capability-card.sh (factory-only gate, "Playground tests").
# One basename per line, exact match (the script reduces both sides to a basename,
# like monitoring/workflow-skill-exempt.txt). Lines starting with # are ignored.
#
# Two legitimate reasons to be on this list:
#   (a) PRE-GATE BASELINE — the workflow already shipped before this gate existed
#       (the 25 below). They are grandfathered, not re-litigated. This is a
#       one-time baseline; do NOT extend it by "matching the pattern".
#   (b) NO NEW EXTERNAL CAPABILITY — a genuinely new workflow that is pure
#       plumbing / orchestration / a thin wrapper over an already-proven primitive
#       (a cron, a sink, a router, an Execute-Workflow sub). Adding such an entry
#       is a conscious "this adds no new verb to prove" decision — say why in a
#       comment above it.
#
# A NEW workflow that DOES add an external capability (a new verb: read / fill /
# extract / send / parse against a real external system) is NOT exempt — it needs
# docs/capability-cards/<name>.md with a go/no-go verdict (see that dir's README
# and templates/agent-design-spec.md §0). That is the whole point of the gate.
